知行者到底是什么系统

知行者不是一个单点 Demo。它是一个个人 AI 知识与行动助手：把你的文档变成知识库，让 AI 基于知识库回答，再把回答转成任务、计划、复盘和学习路线。你现在重点学习的是其中最值钱、最企业化的一段：RAG。

给用户看的能力

创建知识库、上传文档、查看文档、聊天问答、看引用、生成任务、查看趋势资料、运行 RAG 实验室和自动评测。

给工程师看的能力

FastAPI 接口、SQLite 数据表、FAISS 向量检索、本地 Embedding、BM25 混合检索、rerank、质量评估、API Key 加密保存。

给测试工程师看的能力

你可以设计评测集，验证“该答的时候能答、不该答的时候拒答、引用是否命中、参数变化是否让效果变差”。这就是 AI 应用测试。

一句话：知行者的学习价值不是“做一个网页”，而是把 AI 应用开发最常见的工程链路完整跑通。你后面做企业知识库、智能客服、测试知识助手、产品问答系统，底层都是这套思路。
核心概念

RAG 是怎么工作的

RAG 的英文是 Retrieval-Augmented Generation，中文常叫检索增强生成。它的本质不是训练一个新大模型，而是先检索你的资料，再让大模型基于资料回答。

1文档PDF、Word、Markdown、网页、测试文档、产品手册。
2解析把文件里的内容提取成纯文本，去掉无法处理的格式。
3清洗去掉多余空行、乱码、页眉页脚、广告导航。
4切片把长文档切成多个小片段，方便检索和放进 Prompt。
5向量化用 Embedding 模型把每个片段变成一组数字。
6检索把问题也变成向量，从向量库找相似片段。
7生成把命中片段放进 Prompt，让大模型只按依据回答。
8评估检查有没有引用、证据强不强、是否应该拒答。
学习收敛：RAG 名词很多，但你现在只按四条主线学：Naive RAG 跑通基础链路，Advanced RAG 做企业级优化，Agentic RAG 做自主决策，Graph RAG 做复杂关系推理。

Naive RAG：基础 / 标准 RAG

流程是分块、向量化、向量检索、拼接上下文、生成回答。它最简单，适合 FAQ 和简单文档问答，是你必须先跑通的地基。缺点是检索不准时容易答偏，也缺少重排序、评测和拒答保护。

Advanced RAG：进阶 / 企业级 RAG

在 Naive 基础上做全流程优化。检索前做查询重写、扩展、意图识别；检索中做混合检索、多路召回；检索后做 Rerank、上下文压缩、去冗余。它是你当前最该重点深入的方向。

Agentic RAG：智能体 RAG

让大模型动态决定是否检索、检索几次、调用什么工具、证据够不够、是否需要反思修正。它适合复杂多跳问题、长任务、测试风险分析和自动生成任务，是知行者后续变成行动助手的关键。

Graph RAG：图谱 RAG

把文档构建成实体和关系组成的知识图谱，支持多跳关联推理。它适合法律、金融、医疗、论文关系分析等复杂场景，但学习成本高，建议作为后期专项。

入口

文档解析：AI 读资料的第一关

RAG 的第一步不是模型，而是把文件变成干净文本。很多 RAG 系统效果差，不是因为模型差，而是因为文档解析阶段已经把内容弄丢了。

txt / md

最容易处理，通常直接读取文本。Markdown 还能保留标题、列表、代码块，这对 RAG 很友好。

PDF

文本型 PDF 可以提取文字；扫描件 PDF 本质是图片，需要 OCR。表格、双栏论文、页眉页脚常常会干扰结果。

Word docx

可以从 XML 里读取段落文本。难点是表格、图片、批注、页眉页脚和复杂排版。

在知行者里对应哪里
上传接口在 backend/app/main.py 的 /api/knowledge-bases/{id}/documents。文件类型检查、PDF 读取、Word 读取、文本清洗在 backend/app/services.py。上传后，文档记录写入 documents 表，切片写入 document_chunks 表。
0 基础要怎么判断解析是否成功
看三个点：第一，文档状态是不是 ready；第二，打开文档详情，正文是不是完整可读；第三，问一个文档里一定有答案的问题，看检索片段是否命中正确段落。如果 PDF 上传后为空，通常是扫描件，需要 OCR。
RAG 基础功

文本切片：把长文档切成可检索的证据

大模型不能每次都吃完整个知识库，所以要把文档切成片段。切片决定了检索的颗粒度，也决定了回答时拿到的上下文是否完整。

chunk_size 是什么

每个片段大约多长。切得太小，命中更精准，但可能只拿到半句话；切得太大，信息完整，但相似度容易被无关内容稀释。

overlap 是什么

相邻片段之间重复多少内容。它用来避免重要句子刚好被切断。重叠太大也会带来重复片段、更多向量和更高成本。

问题类型建议切片原因你怎么验证
概念解释300 到 600 字符问题通常只需要一小段定义。看命中片段是否包含完整定义。
流程总结600 到 1200 字符流程需要上下文，太短会断。看 1 到 2 个片段能否覆盖完整步骤。
制度问答500 到 900 字符条款通常按段落组织。看引用是否落到对应条款。
代码/配置尽量按代码块切代码被切断会失去可运行性。看片段是否包含完整函数或配置块。
语义变数字

Embedding：让文本可以被计算

Embedding 是把文字变成数字向量。它不负责回答问题，只负责表达“语义位置”。语义相近的文本，向量距离应该更近。

外部 Embedding API

调用云服务生成向量。优点是效果稳定、接入快；缺点是可能收费、需要联网、文档内容会发给外部。

本地 Embedding 模型

知行者当前可用 BAAI/bge-small-zh-v1.5。优点是数据不出本地，适合学习原理；缺点是首次下载模型，运行会占内存。

local-hash 兜底

这是为了让流程离线跑通的简单向量方法，不等于真正语义模型。它适合教学，不适合追求最终效果。

必须分清：Embedding 模型是“把文本变向量”的模型；聊天大模型是“根据上下文生成回答”的模型。RAG 里这两个模型可以不是同一个。
存储与搜索

向量数据库：保存证据的语义坐标

向量数据库不是神秘东西。你可以先理解成：每个文档片段都有一串数字，数据库负责保存这些数字，并快速找出和问题最像的片段。

知行者当前方案

SQLite 保存业务数据、文档、切片、向量 JSON 和配置；FAISS 负责向量相似度计算。这个组合适合本地学习和单机项目。

为什么不一开始就上复杂数据库

0 基础阶段最重要的是看懂链路。FAISS + SQLite 足够你学习解析、切片、向量化、检索、评测。等规模变大，再学 Chroma、Milvus、Qdrant、Pinecone。

文档片段 content ↓ embed_text() 向量 [0.12, -0.03, 0.88, ...] ↓ upsert_chunk_vector() SQLite chunk_vectors 表 ↓ search_similar_chunks() FAISS IndexFlatIP 找最相近片段
找到证据

检索、BM25、Hybrid、Rerank

企业 RAG 不是只做一次向量搜索就结束。真实系统会组合多种信号，让“语义相似”和“关键词命中”互相补位，再重新排序。

向量检索

擅长语义相近。例如“文本切片”和“chunking”字面不同，但语义接近。

BM25

经典关键词检索。擅长专有名词、版本号、错误码、API 名称。

Hybrid

混合检索。知行者当前用向量分 0.65 + BM25 分 0.35 做融合。

Rerank

对候选片段重新排序。知行者会考虑基础分、问题覆盖率和片段完整度。

hybrid_score = vector_score * 0.65 + bm25_score * 0.35 rerank_score = base_score * 0.65 + token_coverage * 0.25 + completeness * 0.10
让模型说人话

生成回答：Prompt、引用和本地兜底

检索到证据后，系统会把问题和证据一起交给大模型。好的 RAG Prompt 会要求模型只基于证据回答，并展示引用来源。

Prompt 里放什么

用户问题、检索片段、引用编号、回答规则。规则里要强调：资料不足时不要编，应该说明没有找到依据。

引用为什么重要

引用让用户能回到原文判断答案是否可信。没有引用的企业问答，很难进入真实业务。

无 Key 也能学习

知行者在没有大模型 API Key 时可以用本地模拟回答，让你先学习 RAG 链路。真正上线时再接 DeepSeek、千问或 OpenAI-compatible API。

测试工程师优势区

评测与拒答：AI 应用不只看“能不能答”

普通 Demo 只证明“能回答”。真实系统要证明“答得有依据、弱依据会提醒、没依据会拒答”。这正是测试工程师转 AI 应用测试的核心竞争力。

Grounded

依据充分。检索片段覆盖了问题中的关键概念，相似度也达到阈值。

Weak Evidence

有一点依据，但不够强。系统应该提醒用户补文档、换问法、调参数，而不是自信胡说。

No Evidence

没有依据。正确行为是拒答或说明当前知识库没有找到资料。

为什么“有没有给出 Kubernetes YAML？”要严格判断
因为这类问题不是泛泛问“有没有提到 Kubernetes”，而是在问文档里是否真的有 YAML 配置。即使检索片段里出现 Kubernetes，如果没有 apiVersion:、kind: 这种直接证据，也应该判定为没有依据。这叫严格无依据判断。
知行者的自动评测集怎么用
你可以添加问题、预期结论和必须命中的关键词。然后用当前 RAG 参数批量运行。现在 Demo 知识库有 35 条用例，检查 RAG 是否稳定。以后你上传公司文档，也应该建立自己的评测集。

真实项目意识

安全、配置和上线前要知道的事

AI 应用里最容易被忽视的是安全：API Key 不能泄露，用户文档可能敏感，模型回答不能越过资料边界。

API Key

应该保存在后端，不应该写死在前端。知行者会对模型配置里的 Key 做加密保存和脱敏展示。

数据边界

本地 Embedding 让文档不离开电脑。外部 API 方便，但要知道哪些内容会发出去。

拒答策略

资料不足时拒答不是功能缺陷，而是可信 AI 的必要能力。企业系统宁愿说不知道，也不能编。

照着做

你的学习路线

你现在不需要先学深度学习算法。先把 AI 应用工程跑通，再补算法和模型原理。测试工程师的优势是验证、质量、边界条件和自动化评测。

第 1 阶段：跑通系统

  1. 启动后端，打开前端。
  2. 创建知识库，上传 Markdown、PDF、Word。
  3. 查看文档详情，确认解析文本正常。
  4. 问一个有依据的问题，看引用是否正确。

第 2 阶段：理解 RAG

  1. 进入 RAG 实验室。
  2. 固定同一个问题，只改 chunk_size。
  3. 固定同一个问题，只改 overlap。
  4. 对比开启和关闭 hybrid、rerank。

第 3 阶段：学会评测

  1. 添加 grounded 用例。
  2. 添加 no_evidence 用例。
  3. 运行评测集，记录通过率。
  4. 改参数后重新跑，看是否退化。

第 4 阶段：升级系统

  1. 接入一个真实聊天模型 API。
  2. 尝试不同 Embedding 模型。
  3. 增加网页抓取或 OCR。
  4. 把评测脚本接入自动化。
