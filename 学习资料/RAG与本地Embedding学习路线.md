# RAG 与本地 Embedding 学习路线

## 1. 你现在到底要学什么

你要学的不是单独某一个工具，而是一套企业最常见的 AI 应用工程能力：

```text
文档 -> 切片 -> Embedding -> 向量数据库 -> 检索 -> Prompt -> 大模型回答 -> 引用 -> 质量评估
```

这套能力叫 RAG，中文通常叫“检索增强生成”。

它解决的问题是：让 AI 基于你的文档回答，而不是凭空编。

## 2. 一句话理解 RAG

RAG 的本质是：

```text
先从知识库里找证据，再让大模型基于证据回答。
```

普通聊天是：

```text
用户问题 -> 大模型直接回答
```

RAG 是：

```text
用户问题 -> 检索相关文档片段 -> 把片段交给大模型 -> 基于片段回答
```

所以 RAG 的关键不是“模型多聪明”，而是“证据找得准不准”。

## 3. 你应该掌握的核心模块

### 3.1 文档解析

你要知道不同文件怎么变成文本：

- txt / md：直接读取。
- PDF：文本型 PDF 可以提取，扫描件需要 OCR。
- Word：提取正文段落。
- 网页：提取正文，去掉导航、广告、页脚。

你要练习：

- 上传不同格式文档。
- 看解析出来的文本是否完整。
- 检查标题、表格、段落有没有丢失。

做到什么程度算会：

- 你能判断一个 PDF 为什么提取不到文字。
- 你知道扫描件要先 OCR。
- 你知道表格、图片、批注是文档解析里的难点。

### 3.2 文本清洗

解析后的文本经常有噪声：

- 多余空行。
- 页眉页脚。
- 重复目录。
- 网页导航。
- 乱码。

你要练习：

- 看上传后的正文内容。
- 判断哪些内容会干扰检索。
- 尝试手动改文档内容后重新保存。

做到什么程度算会：

- 你知道“垃圾文本进知识库，RAG 结果也会变差”。
- 你能说明为什么清洗比想象中重要。

### 3.3 文本切片 Chunking

切片是 RAG 最重要的基础能力之一。

大文档不能整篇塞给模型，所以要拆成很多小片段。

常见参数：

```text
chunk_size：每个片段多长
overlap：相邻片段重复多少
```

经验：

- 切太小：检索更聚焦，但容易丢上下文。
- 切太大：上下文完整，但相似度可能变钝。
- 有 overlap：能缓解句子被切断的问题。
- overlap 太大：片段重复变多，成本上升。

你要练习：

在 RAG 实验室里，同一个问题只改一个参数：

```text
chunk_size = 300 / 600 / 900 / 1500
overlap = 0 / 80 / 160 / 300
```

观察：

- 临时切片数量有没有变化。
- 命中的片段是否完整。
- 相关度分数是否变化。
- 是否把答案切断了。

做到什么程度算会：

- 你能解释为什么“流程总结类问题”适合较大切片。
- 你能解释为什么“单点概念类问题”可能适合较小切片。
- 你知道不同文档没有万能参数。

### 3.4 Embedding

Embedding 是把文本变成数字向量。

简单理解：

```text
文本 -> 一组数字 -> 可以被计算相似度
```

比如：

```text
“文本切片” 和 “chunking” 的向量应该比较接近
```

Embedding 不负责回答问题，它只负责表达语义。

你要知道三种模式：

#### 外部 Embedding API

```text
文本 -> 阿里/硅基/OpenAI API -> 向量
```

优点：

- 接入快。
- 效果稳定。
- 本地压力小。

缺点：

- 需要联网。
- 可能收费。
- 文档内容会发给外部服务。

#### 本地 Embedding 模型

```text
文本 -> 本地 bge-small-zh-v1.5 -> 向量
```

优点：

- 数据不出本地。
- 不依赖外部 Embedding API。
- 更适合学习底层原理。

缺点：

- 第一次需要下载模型。
- 本地会占用内存和磁盘。
- 比 API 慢一些。

#### 本地 hash 兜底

你的项目当前有：

```text
local-hash-384
```

它可以让流程跑通，但它不是真正的语义 Embedding。

你要知道：

- 它适合学习流程。
- 它不适合追求最终检索效果。
- 后续应该升级成本地模型或真实 Embedding API。

做到什么程度算会：

- 你能说清楚 Embedding 和大模型回答不是一回事。
- 你知道 Embedding 是为了检索，不是为了聊天。
- 你知道真实 Embedding 会比 hash 兜底更准。

### 3.5 向量数据库

向量数据库负责存向量和查相似向量。

你的项目现在用：

```text
FAISS + SQLite
```

含义：

- FAISS：负责向量相似度检索。
- SQLite：保存文档、切片、向量 JSON、元数据。

当前对比方式：

```text
FAISS IndexFlatIP
```

也就是内积相似度。

因为向量会先归一化，所以可以近似理解为：

```text
余弦相似度
```

常见向量库：

- FAISS：适合本地学习和单机检索。
- Chroma：适合快速搭 RAG Demo。
- Qdrant：适合服务化向量数据库。
- pgvector：适合把向量存在 PostgreSQL。
- Milvus：适合更大规模向量检索。

做到什么程度算会：

- 你知道向量数据库不是大模型。
- 你知道它负责“找相似片段”。
- 你能解释 FAISS 和 SQLite 在项目里分别负责什么。

### 3.6 检索 Top K

Top K 是每次返回几个最相关片段。

```text
top_k = 3：更聚焦，可能漏证据
top_k = 5：常用默认值
top_k = 8：召回更多，但可能混入噪声
```

你要练习：

同一个问题分别跑：

```text
top_k = 1
top_k = 3
top_k = 5
top_k = 8
```

观察：

- 第一条是否最准确。
- 后面的片段是不是补充证据。
- 是否出现无关片段。

做到什么程度算会：

- 你知道“召回多”不一定更好。
- 你知道 top_k 太小会漏信息，太大会带噪声。

### 3.7 Rerank 重排序

Embedding 检索第一轮可能找得不够准。

Rerank 是第二轮排序：

```text
先召回 20 个片段 -> rerank 重新排序 -> 取前 5 个给大模型
```

它解决的问题：

- 关键词命中了，但片段不一定真的能回答。
- 向量相似度高，但证据可能不完整。
- 第一条不一定是最好的引用。

知行者当前已经加入了轻量 Rerank：

```text
先用向量检索召回更多候选片段
再综合向量分、问题词覆盖、片段完整度重新排序
最后取 top_k 个片段给问答链路
```

它不是最强的模型版 rerank，但非常适合学习：

- 为什么“召回”和“排序”是两步。
- 为什么相似度最高的片段不一定最好。
- 为什么完整片段更适合当引用。

做到什么程度算会：

- 你能解释为什么“检索命中”不等于“引用正确”。
- 你知道 rerank 是为了提高证据排序质量。
- 你能在 RAG 实验室里开关 rerank，对比排序变化。

### 3.8 Prompt 拼接

检索到片段后，要把片段放进提示词里。

典型 Prompt：

```text
你是知识库问答助手。
请只基于以下参考资料回答。
如果资料不足，请明确说明没有找到依据。

用户问题：
...

参考资料：
[1] ...
[2] ...
```

重点：

- 不要让模型脱离资料乱答。
- 要求模型说明依据不足。
- 要求模型不要编造引用。

做到什么程度算会：

- 你知道 RAG 不是只检索，还要设计 Prompt。
- 你知道 Prompt 要约束模型“基于证据回答”。

### 3.9 引用来源

企业 RAG 一定要展示引用。

引用的价值：

- 用户能验证答案。
- 测试人员能判断答案是否有依据。
- 产品能降低幻觉风险。

你要练习：

- 看回答引用的是不是正确文档。
- 看引用片段是否真的支持答案。
- 看无依据问题是否拒答。

做到什么程度算会：

- 你能发现“答案看起来对，但引用不支持”的问题。
- 你能把引用正确性写成测试用例。

### 3.10 RAG 质量评估

你是测试工程师，这里是你的优势。

你要重点评估：

- 检索命中率。
- 引用正确性。
- 无依据拒答。
- 答案是否只基于资料。
- 不同参数下的结果差异。
- 模型失败时是否有兜底。

常见测试问题类型：

| 类型 | 示例 | 预期 |
| --- | --- | --- |
| 明确事实 | RAG 的核心步骤是什么？ | 命中包含步骤的片段 |
| 局部概念 | Embedding 起什么作用？ | 命中向量化相关片段 |
| 总结归纳 | 这篇文档主要讲什么？ | 命中多个核心片段 |
| 无依据 | 文档里有没有 Kubernetes 部署方案？ | 没依据时拒答 |
| 边界问题 | 问跨段落的信息 | overlap 可能改善命中 |

做到什么程度算会：

- 你能设计一组 RAG 回归测试集。
- 你能判断一次 RAG 改动是变好还是变差。

### 第 6 阶段：学习 RAG 相关算法基础

目标：

```text

```

你不需要一开始学很深的算法竞赛内容。你应该优先学习和 AI 应用直接相关的算法。

#### 6.1 向量基础

你要理解：

- 什么是向量。
- 文本为什么可以表示成向量。
- 向量维度是什么意思。
- 为什么两个文本可以通过向量距离比较相似。

要会解释：

```text
Embedding 模型把文本变成一串数字，这串数字就是向量。
语义相近的文本，向量距离通常更近。
```

练习：

- 找两个相似问题，看它们是否命中同一片段。
- 找两个完全无关问题，看相似度是否下降。

#### 6.2 相似度算法

你要学：

- 余弦相似度 cosine similarity。
- 内积 inner product。
- 欧氏距离 Euclidean distance。
- 向量归一化 normalization。

在知行者里，当前 FAISS 使用：

```text
IndexFlatIP
```

也就是内积相似度。

因为向量会做归一化，所以可以近似理解为：

```text
归一化后的内积 ≈ 余弦相似度
```

做到什么程度算会：

- 你能解释“相似度分数越高，代表问题和片段越接近”。
- 你知道 0.8 不一定永远好，分数要结合模型和数据分布看。
- 你知道不同 embedding 模型的分数不能随便横向比较。

#### 6.3 Top K 检索

你要学：

- Top K 是什么。
- 为什么召回多个片段。
- top_k 太小和太大的风险。

经验：

```text
top_k 太小：可能漏掉正确证据。
top_k 太大：可能混入无关内容，让模型被干扰。
```

练习：

在 RAG 实验室里同一问题分别设置：

```text
top_k = 1
top_k = 3
top_k = 5
top_k = 8
```

观察第一条是否正确，后面的片段是否有帮助。

#### 6.4 关键词检索与倒排索引

你要理解传统搜索是怎么做的：

- 分词。
- 关键词匹配。
- 倒排索引。
- TF-IDF。
- BM25。

简单理解：

```text
倒排索引像一本词典：
某个词出现在哪些文档里，系统提前记下来。
```

BM25 是很多搜索系统的经典排序算法，它比简单关键词匹配更稳。

为什么要学：

- RAG 不一定只用向量检索。
- 企业里经常使用“关键词检索 + 向量检索”的混合检索。
- 有些专有名词、编号、接口名、错误码，用关键词检索反而更准。

做到什么程度算会：

- 你知道关键词检索适合精确词。
- 你知道向量检索适合语义相似。
- 你能解释为什么混合检索比单一路线更稳。

#### 6.5 Embedding 与语义空间

你要学：

- 词向量。
- 句向量。
- 语义空间。
- 中文 embedding 模型。
- 多语言 embedding 模型。

你不需要训练模型，但要知道模型在做什么：

```text
它把文本压缩成向量，让相似语义在空间里更接近。
```

你要重点对比：

```text
local-hash-384
bge-small-zh-v1.5

```

做到什么程度算会：

- 你知道中文文档优先选中文或多语言 embedding。
- 你知道模型变了，历史向量需要重新生成。

聚类可以用来：

分类可以用来：

你要先理解概念，不用急着手写算法。

#### 6.7 Rerank 重排序算法

你要理解：

```text
第一轮检索负责“召回”，rerank 负责“重新排序”。
```

为什么需要 rerank：

- 向量相似度高，不代表片段真的能回答问题。
- 有些片段只是出现关键词，但证据不完整。
- top_k 很大时，需要二次筛选最好的证据。

典型流程：

```text
向量检索召回 20 条 -> rerank 重新排序 -> 取前 5 条给大模型
```

知行者当前采用的是轻量可解释 rerank，公式可以简单理解为：

```text
rerank 分 = 向量相似度 * 0.65 + 问题词覆盖 * 0.25 + 片段完整度 * 0.10
```

这三个分数分别代表：

- 向量相似度：语义上像不像。
- 问题词覆盖：问题里的关键字有没有被片段覆盖。
- 片段完整度：片段是否像一段完整证据，而不是被切断的半句话。

为什么先用这种轻量方案：

- 不需要再下载新的 rerank 模型。
- 能直接看懂排序为什么变化。
- 适合你先学习“召回后重排”的工程思想。

以后可以升级成模型版 rerank：

```text
query + chunk -> rerank 模型 -> 更准确的相关性分数
```

做到什么程度算会：

- 你能解释为什么“命中片段”不等于“命中正确片段”。
- 你知道 rerank 是企业 RAG 的常见增强手段。
- 你能看懂 RAG 实验室里的向量分、rerank 分和覆盖率。

#### 6.8 RAG 评估指标

你作为测试工程师，最应该重视这一块。

你要学：

- Hit Rate：是否命中正确文档。
- Recall@K：前 K 个结果里是否包含正确证据。
- Precision@K：前 K 个结果里有多少是真的相关。
- MRR：正确答案排得越靠前越好。
- NDCG：排序质量指标。
- 引用正确率：引用是否真的支持答案。
- 无依据拒答率：知识库没有资料时是否拒答。
- 幻觉率：模型是否编造内容。

你不用一开始背公式，先会用业务语言解释。

例子：

```text
如果正确片段排在第 1 名，比排在第 5 名更好。
如果 top_k=5 里有 3 个无关片段，说明 Precision@5 不高。
如果文档里没有答案但模型仍然回答，说明无依据拒答失败。
```

做到什么程度算会：

- 你能写一份 RAG 测试用例表。
- 你能判断一次参数调整是让系统变好还是变差。
- 你能发现错引、漏召回、幻觉这三类问题。

## 18. RAG 分类怎么学：只保留四条主线

RAG 的分类很多，但你现在不要把 Standard、Hybrid、Adaptive、Fusion、Multi-Modal、Structured、Code、Conversational 全部当成并列主线学习。

更适合你的学习方式是只抓四条主线：

```text
Naive RAG -> Advanced RAG -> Agentic RAG -> Graph RAG
```

其他分类先归类理解：

- Standard RAG：基本归入 Naive RAG。
- Hybrid RAG、Fusion RAG、Adaptive RAG：多数能力归入 Advanced RAG。
- Multi-Modal RAG、Structured RAG、Code RAG：属于知识形态或应用对象扩展，后面做专项。
- Conversational RAG：属于对话形态扩展，可以放在 Advanced 或 Agentic 阶段学习。

### 18.1 Naive RAG：基础 / 标准 RAG

流程：

```text
分块 -> 向量化 -> 向量检索 -> 拼接上下文 -> 生成
```

特点：

- 最简单。
- 易部署。
- 最适合入门。

缺点：

- 检索不准时容易答偏。
- 没有重排序和评测时容易幻觉。

适用：

- FAQ。
- 简单文档问答。
- RAG 学习 Demo。

你在知行者里要做：

```text
上传文档 -> 看切片 -> 跑 RAG 实验室 -> 看引用是否命中
```

### 18.2 Advanced RAG：进阶 / 企业级 RAG

Advanced RAG 是你现在最应该重点学的部分。

它在 Naive 基础上做全流程优化：

检索前：

- 查询重写。
- 查询扩展。
- 意图识别。

检索中：

- 向量 + BM25 混合检索。
- 多路召回。
- 不同索引融合。

检索后：

- Rerank 重排序。
- 上下文压缩。
- 去冗余。
- 引用筛选。

适用：

- 企业知识库。
- 测试知识助手。
- 智能客服。
- 产品助手。

知行者已经具备的 Advanced RAG 能力：

- Hybrid 检索。
- BM25。
- Rerank。
- 无依据拒答。
- RAG 实验室。
- 自动评测集。

你在知行者里要做：

```text
同一个问题只改一个变量：
1. 开关 Hybrid
2. 开关 Rerank
3. 调整 Top K
4. 调整 chunk_size
5. 跑自动评测集
```

### 18.3 Agentic RAG：智能体 RAG

Agentic RAG 是让大模型动态决定：

- 是否需要检索。
- 检索哪个知识库。
- 检索几次。
- 是否调用工具。
- 证据是否足够。
- 是否反思修正。

适用：

- 复杂多跳问题。
- 长任务。
- 自动分析报告。
- 测试风险分析。
- 需要工具调用的工作流。

它的缺点：

- 更难测试。
- 容易循环。
- 成本更难控制。
- 必须有步骤日志和停止条件。

知行者未来可以这样做：

```text
用户提出目标
-> Agent 判断需要哪些资料
-> 检索知识库
-> 判断依据是否足够
-> 不够就继续检索或要求补文档
-> 生成回答
-> 生成任务
-> 进入复盘
```

### 18.4 Graph RAG：图谱 RAG

Graph RAG 是把文档构建成知识图谱：

```text
实体 + 关系
```

例子：

```text
RAG -> 包含 -> Embedding
Embedding -> 用于 -> 语义检索
BM25 -> 用于 -> 关键词检索
Rerank -> 用于 -> 结果重排序
```

适用：

- 金融风控。
- 医疗知识关联。
- 法律条文推理。
- 论文关系分析。
- 组织关系和项目依赖分析。

它的缺点：

- 构建成本高。
- 实体抽取和关系抽取容易错。
- 需要图数据库或图查询能力。
- 不适合现在作为第一主线。

你现在只需要先做到：

```text
能从文档里识别实体和关系，并画出简单关系图。
```

你的路线应该是：

```text

第 2 步：Advanced RAG

第 3 步：Agentic RAG

第 4 步：Graph RAG
目标：做实体关系、多跳推理、复杂知识关联。
```

