## 1. 当前 AI 工程最值得关注的方向

### 1.1 AI Agent 和 Coding Agent 正在快速升温

GitHub Trending 和 OSSInsight 的 AI 排行都显示，AI Agent、Coding Agent、MCP、RAG、向量数据库、推理框架是 2026 年开源生态的核心热点。

OSSInsight 的 AI 趋势页把项目分成 AI Agents、LLM Tools、MCP Servers、Coding Agents、RAG、Inference、Vector DB、Vibe Coding、AI Assistants 等类别。它还显示 Top Movers 包括 opencode、claude-code、codex、llama.cpp、open-webui 等项目。

GitHub Trending 当天也出现了多个 Agent 相关项目，例如：

- DeepSeek-TUI：面向 DeepSeek 模型的终端编码 Agent。
- ruflo：面向 Claude 的多智能体编排平台，强调 swarm、RAG integration、Claude Code / Codex Integration。
- dexter：面向深度金融研究的 autonomous agent。
- context-mode：面向 AI coding agents 的上下文窗口优化工具。
- cocoindex：面向 long horizon agents 的增量索引引擎。
- awesome-ai-apps：收集 RAG、agents、workflows 等 AI 应用案例。
- local-deep-research：支持本地和云端 LLM、多搜索源和私有文档的研究工具。

学习重点：

- Agent 不是只会聊天，而是会拆任务、调用工具、保存上下文、执行流程。
- Coding Agent 会影响软件开发、测试、代码审查、需求拆解和自动化修复。
- RAG 经常作为 Agent 的记忆、知识检索、文档依据层。
- MCP 类工具在解决 Agent 与外部系统连接的问题。

你可以重点问知行者：

- 2026 年 AI Agent 开源生态有哪些方向？
- Coding Agent 和传统代码补全有什么区别？
- RAG 为什么会成为 Agent 的基础能力？
- MCP Server 在 Agent 系统中解决什么问题？

来源：

- OSSInsight AI Trending: https://ossinsight.io/trending/ai
- GitHub Trending: https://github.com/trending

## 2. GitHub 上适合学习的开源方向

### 2.1 RAG 框架和知识库工具

值得关注的项目类别：

- RAGFlow：偏企业知识库、文档解析、问答流程。
- AnythingLLM：偏本地/私有知识库和 LLM 应用。
- LlamaIndex：偏数据连接器、索引、检索和 RAG 工程链路。
- LangChain：偏 LLM 应用开发、Agent、工具调用、链式编排。
- Dify：偏低代码 LLM 应用平台，适合理解产品化 AI 工作流。

你现在做的知行者已经覆盖了其中的入门链路：

```text
文档上传 -> 解析 -> 切片 -> Embedding -> FAISS -> 混合检索 -> Rerank -> 引用 -> 评测
```

下一步可以对照这些开源项目学习：

- 它们如何管理知识库。
- 它们如何做文档解析和切片。
- 它们如何做 RAG 评测。
- 它们如何展示引用来源。
- 它们如何处理无依据拒答。

### 2.2 向量数据库和本地检索

值得关注的项目类别：

- FAISS：适合单机、本地学习、向量索引实验。
- Qdrant：适合服务化向量数据库。
- Chroma：适合快速搭建 RAG Demo。
- Milvus：适合更大规模向量检索。

你现在的知行者使用：

```text
SQLite 保存文档和元数据
FAISS 做本地向量检索
BM25 做关键词检索补充
```

这已经非常适合学习 RAG 工程基础。等你掌握流程后，可以再切到 Qdrant 或 Chroma 做对比。

### 2.3 AI Coding Agent 和测试工程师转型

值得关注的项目类别：

- OpenHands：开源软件工程 Agent。
- Codex：面向代码任务的 Agent。
- Claude Code：面向代码编辑、项目理解和自动化开发。
- Cline、Aider、Continue：面向 IDE 或命令行的 AI 编程助手。

测试工程师学习这些项目时，不应该只看“能不能写代码”，更应该看：

- 它如何理解需求。
- 它如何定位 bug。
- 它如何生成测试。
- 它如何避免误改。
- 它如何解释改动影响范围。
- 它如何做回归验证。

这和你现在在知行者里做 RAG 自动评测集的思路是相通的：

```text
AI 生成结果并不等于正确
必须有评测集、引用、回归、失败原因
```

## 3. Hugging Face / arXiv 上值得学习的论文方向

### 3.1 HERA：Multi-agent RAG with Evolving Orchestration and Agent Prompts

论文：Experience as a Compass: Multi-agent RAG with Evolving Orchestration and Agent Prompts

发布时间：2026-04-02

核心问题：

传统 Multi-agent RAG 往往使用固定 Agent 行为和固定编排策略，面对复杂多跳问题时容易变脆弱。

核心思路：

- 使用 HERA 这样的层级框架。
- 全局层优化 query-specific agent topologies。
- 局部层优化 role-specific prompts。
- 通过经验积累和奖励引导，让多智能体编排逐渐适应不同问题。

对你的启发：

- RAG 不只是“检索几个片段给模型”。
- 更复杂的问题可能需要多个 Agent 分角色协作。
- Agent 的 prompt 和编排方式也可以被评测和优化。
- 你后续可以把知行者扩展成“检索 Agent + 总结 Agent + 测试 Agent + 任务拆解 Agent”。

建议提问：

- HERA 解决了 Multi-agent RAG 的什么问题？
- 多智能体 RAG 和普通 RAG 有什么区别？
- Agent 编排为什么需要动态调整？

来源：

- Hugging Face Papers: https://huggingface.co/papers/2604.00901

### 3.2 MiRAGE：面向 RAG 评测集生成的多智能体框架

论文：MiRAGE: A Multiagent Framework for Generating Multimodal Multihop Question-Answer Dataset for RAG Evaluation

提交时间：2026-01-21

核心问题：

企业 RAG 已经走向多模态、高风险、专业文档场景，但领域专属评测基准不足。传统数据集往往太通用，无法覆盖复杂技术文档、多跳推理和多模态证据。

核心思路：

- 用多智能体协作生成 RAG 评测数据集。
- 通过递归上下文优化聚合分散证据。
- 使用 adversarial verifier 保证事实依据。
- 识别专家角色和领域，模拟专家工作方式。

对你的启发：

- 你现在做的 RAG 自动评测集是正确方向。
- 企业级 RAG 不是只看回答是否通顺，而是要评估引用、事实、召回、多跳证据。
- 后续可以让知行者自动从文档中生成评测问题，再由你审核。

建议提问：

- MiRAGE 为什么要用多智能体生成 RAG 评测集？
- RAG 评测为什么需要多跳问题？
- 企业知识库为什么不能只用通用评测数据？

来源：

- arXiv: https://arxiv.org/abs/2601.15487

### 3.3 AIDev：研究 GitHub 上 AI Coding Agent 的真实使用

论文：AIDev: Studying AI Coding Agents on GitHub

提交时间：2026-02-09

核心问题：

AI Coding Agent 正在改变软件工程，但研究社区缺少大规模真实数据来分析 Agent 如何参与真实项目。

核心数据：

- AIDev 收集了 932,791 个 Agent-authored Pull Requests。
- 涉及 OpenAI Codex、Devin、GitHub Copilot、Cursor、Claude Code 等 5 类 Agent。
- 覆盖 116,211 个仓库和 72,189 名开发者。
- 还包含一个更细的子集：33,596 个 Agentic PR，来自 2,807 个 100 stars 以上仓库。

对你的启发：

- Coding Agent 已经不是玩具，而是在真实 GitHub 工作流中生成 PR。
- 测试工程师未来需要学会评估 Agent 生成的代码、测试、修复和 PR 风险。
- 你可以把“AI 生成代码是否正确”看成一种测试对象。

建议提问：

- AIDev 数据集研究了哪些 AI Coding Agent？
- Agent-authored PR 对测试工程师意味着什么？
- 为什么 AI Coding Agent 需要回归测试和代码审查？

来源：

- arXiv: https://arxiv.org/abs/2602.09185

### 3.4 AI Model Cards：动态更新 AI 文档，提高模型可复用性

论文：Toward Reusability of AI Models Using Dynamic Updates of AI Documentation

提交时间：2026-04-19

核心问题：

很多 AI 模型虽然已经训练好，但因为缺少合适的 AI 文档和 model cards，导致别人难以复用。

核心思路：

- 让模型卡片更敏捷、更数据驱动、更贴近社区实践。
- 使用 Hugging Face 模型仓库作为研究对象之一。
- 关注 AI 文档模板如何跟上快速变化的最佳实践。

对你的启发：

- AI 项目不只是模型和代码，文档质量也直接影响可复用性。
- 未来你做知行者知识库时，也要关注“文档是否可检索、是否结构化、是否能支持评测”。
- 测试工程师可以把 model card 看成 AI 系统的验收资料之一。

建议提问：

- AI model card 为什么影响模型复用？
- Hugging Face 模型文档对企业 AI 项目有什么启发？
- 知识库里的文档应该如何写，才更适合 RAG？

来源：

- arXiv: https://arxiv.org/abs/2604.17626

## 4. 你现在应该怎么学

### 第一步：继续围绕 RAG 做深

你现在已经做了：

```text
本地 Embedding
FAISS 向量数据库
BM25 + 向量混合检索
Rerank
无依据拒答
RAG 自动评测集
```

下一步应该做：

```text
上传更多真实文档 -> 扩充评测集 -> 对比参数 -> 输出质量报告
```

### 第二步：开始理解 Agent

Agent 学习路线：

```text
工具调用 -> 多步骤任务 -> 记忆/知识库 -> 多 Agent 协作 -> 评测与回归
```

你不要一上来就追复杂 Agent 框架。先把一个“能调用工具的知识助手”做好：

- 能检索文档。
- 能基于引用回答。
- 能拒答。
- 能生成任务。
- 能保存评测结果。
- 能解释失败原因。

这比空学框架更有价值。

### 第三步：把测试能力迁移到 AI 应用

你作为测试工程师，可以重点学这些：

- RAG 测试用例设计。
- 引用正确性测试。
- 无依据拒答测试。
- Agent 工具调用测试。
- 模型输出稳定性测试。
- AI 生成代码的回归测试。
- Prompt 改动前后的质量对比。

这是一条很现实的转型路径：

```text
测试工程师 -> AI 应用测试工程师 -> RAG/Agent 质量工程师 -> AI 应用工程师
```

## 6. 一句话总结

2026 年 AI 应用开发的核心不是单纯调大模型，而是：

```text
RAG 提供可靠知识
Agent 执行复杂任务
评测集保证系统质量
文档和引用降低幻觉风险
```

你现在从知行者开始学 RAG，是一条很稳的路线。下一阶段要把它从“能回答”升级成“可评测、可迭代、可解释、可扩展”。
