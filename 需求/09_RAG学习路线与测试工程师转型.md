# RAG 学习路线与测试工程师转型

## 1. 判断结论

知行者适合作为测试工程师学习大模型应用开发的第一个系统项目。

原因：

1. 它已经具备 RAG 主链路：上传文档、解析文本、切片、检索、问答、引用。
2. 它能接入 DeepSeek、千问、MiniMax 等 OpenAI 兼容模型。
3. 它有前端、后端、数据库和真实接口，适合从 Demo 走向系统。
4. 测试工程师可以把优势放在 RAG 质量评估、拒答测试、引用校验和回归测试上。

但它不是最终形态。当前版本已经接入 Embedding + FAISS 本地向量库；如果没有配置真实 embedding API，会使用本地 hash embedding 兜底。后续可继续升级到 Chroma、pgvector 或 Milvus。

## 2. 当前阶段目标

当前阶段不追求“企业级大而全”，只追求把 RAG 核心闭环吃透。

目标闭环：

```text
上传文档 -> 解析文本 -> 文本切片 -> 生成向量 -> 检索片段 -> 大模型回答 -> 展示引用 -> 质量评估
```

当前必须掌握：

- 文档解析：txt、Markdown、文本型 PDF、docx。
- 切片策略：chunk_size、overlap 对检索结果的影响。
- 检索策略：top_k、相关度、无结果处理。
- 回答策略：只基于资料回答，资料不足时拒答。
- 引用策略：每条回答必须能追溯到文档和片段。
- 评估策略：相关问题能召回，无关问题能拒答。

## 3. 测试工程师的优势切入点

普通开发者容易只做出“能回答”的 Demo。测试工程师应该把它做成“可验证、可信任”的系统。

重点能力：

1. 构造黄金问题集。
2. 验证正确文档是否被召回。
3. 验证引用片段是否支持答案。
4. 验证无依据问题是否拒答。
5. 验证模型失败时是否有兜底。
6. 验证不同 chunk_size、top_k、embedding 模型的效果差异。
7. 建立 RAG 回归测试。

## 4. 学习路线

### 第 1 阶段：跑通知行者 RAG MVP

验收标准：

- 能启动前后端。
- 能创建知识库。
- 能上传 txt、md、pdf、docx。
- 能提问并看到引用。
- 能看到 RAG 质量检查。
- 能调用 `/api/rag/evaluate`。

### 第 2 阶段：替换为真实 Embedding

目标：

- 学会调用 embedding API。
- 把文本切片转成真实向量。
- 对比本地 hash embedding 和真实 embedding 的效果。

建议顺序：

1. 先配置 OpenAI 兼容 embedding。
2. 使用现有 FAISS 本地向量库验证检索效果。
3. 再考虑 Chroma、Milvus 或 pgvector。

### 第 3 阶段：做 RAG 自动化测试

目标：

- 建立一批标准问题。
- 每个问题绑定期望文档。
- 自动跑检索。
- 输出命中率和失败案例。

基础指标：

- recall_at_1
- recall_at_3
- citation_required
- no_evidence_rejection
- average_top_score

### 第 4 阶段：企业化能力

目标：

- 文档权限隔离。
- 多知识库检索。
- 检索日志。
- 模型调用日志。
- 成本统计。
- 批量评估报告。

## 5. 下一步最应该做什么

优先级从高到低：

1. 增加 RAG 黄金问题集表和批量评估接口。
2. 配置真实 embedding 模型并对比检索效果。
3. 增加 PDF 页码级引用。
4. 增加回答是否被引用支撑的人工标注入口。
5. 评估是否需要引入 Chroma、pgvector 或 Milvus。

暂时不要急着做：

- 多 Agent。
- 复杂工作流编排。
- 多用户权限系统。
- 知识图谱。
- 微调模型。

这些能力以后有价值，但不是当前学习 RAG 的第一优先级。
