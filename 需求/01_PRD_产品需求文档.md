# 知行者 PRD 产品需求文档

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 知行者 |
| 产品类型 | 个人 AI 知识与行动助手 |
| 文档版本 | V1.0 |
| 适用阶段 | MVP 到 V1 |
| 目标平台 | Web 优先，后续支持移动端 |
| 编写角色 | 产品经理 |

## 2. 背景与问题

现代个人学习和工作中，用户面临的问题不是资料太少，而是资料太多、太散、太难转化为行动。

典型问题包括：

1. 收藏大量文章、PDF、课程资料，但缺少再次利用。
2. 资料分散在网盘、本地文件、笔记软件和聊天记录中。
3. 学习目标很大，但缺少分阶段计划和每日行动。
4. AI 聊天可以临时回答问题，但回答和资料、任务、复盘割裂。
5. 用户知道很多道理，但难以形成持续执行。

知行者希望用 AI 把「知道」和「做到」连接起来。

## 3. 产品目标

### 3.1 业务目标

- 打造一个可持续演进的个人 AI 系统。
- 先完成本地/个人可用版本，再逐步扩展为云端产品。
- 形成知识库、问答、计划、任务、复盘的产品闭环。

### 3.2 用户目标

- 降低资料整理成本。
- 提升学习和工作资料的复用率。
- 帮助用户从资料中快速获得答案。
- 帮助用户把想法和建议转成任务。
- 通过复盘提升长期执行力。

## 4. 用户画像

### 4.1 用户 A：自学者

- 年龄：20-35 岁
- 需求：学习新技术、考证、语言、专业技能
- 痛点：学习资料很多，但路线不清晰，容易半途而废
- 核心功能：资料问答、学习计划、每日任务、复盘

### 4.2 用户 B：知识工作者

- 年龄：25-40 岁
- 需求：处理文档、会议纪要、行业报告、产品资料
- 痛点：查找资料慢，信息复用低
- 核心功能：文档总结、知识库搜索、基于资料问答

### 4.3 用户 C：内容沉淀者

- 年龄：20-45 岁
- 需求：读书、阅读文章、整理观点
- 痛点：看过很多内容，但没有形成自己的知识体系
- 核心功能：摘要、知识卡片、标签、关联推荐

## 5. 用户故事

1. 作为用户，我希望创建不同主题的知识库，以便分类管理资料。
2. 作为用户，我希望上传 PDF、Markdown、txt 文档，以便系统自动解析内容。
3. 作为用户，我希望系统自动总结文档，以便快速了解主要内容。
4. 作为用户，我希望基于某个知识库向 AI 提问，以便获得与资料相关的回答。
5. 作为用户，我希望回答能展示引用来源，以便判断可信度。
6. 作为用户，我希望把 AI 的建议转成任务，以便后续执行。
7. 作为用户，我希望查看任务列表，以便管理自己的行动。
8. 作为用户，我希望记录复盘，以便系统调整计划。
9. 作为用户，我希望配置不同模型 API，以便根据成本和效果切换模型。

## 6. 核心业务流程

### 6.1 知识库问答流程

1. 用户创建知识库。
2. 用户上传文档。
3. 系统解析文档文本。
4. 系统对文本进行分段切片。
5. 系统生成向量并存储。
6. 用户输入问题。
7. 系统检索相关文档片段。
8. 系统拼接上下文并调用大模型。
9. AI 返回答案。
10. 系统展示答案和引用来源。

### 6.2 任务生成流程

1. 用户在问答结果中点击「生成任务」。
2. 系统将 AI 回答传入任务生成提示词。
3. 模型输出结构化任务列表。
4. 用户确认、编辑或删除任务。
5. 系统保存任务。
6. 用户后续完成或延期任务。

### 6.3 学习计划流程

1. 用户输入学习目标、周期和可用时间。
2. 系统读取用户选择的知识库内容摘要。
3. AI 生成阶段计划。
4. 系统拆解为周计划和日任务。
5. 用户执行任务。
6. 系统根据完成情况和复盘内容调整计划。

## 7. 功能范围

### 7.1 MVP 范围

- 用户登录，可先用本地单用户模式替代。
- 知识库增删改查。
- 文档上传，支持 PDF、txt、Markdown。
- 文档解析和切片。
- 基于知识库问答。
- 答案引用来源。
- AI 摘要。
- 从回答生成任务。
- 任务列表管理。
- 模型 API 配置。

- 知识卡片。

## 9. 权限与账号

MVP 阶段可以使用单用户模式，降低开发复杂度。

V1 开始支持账号体系：

- 邮箱注册登录
- 密码登录
- 用户个人空间隔离
- API Key 加密存储

暂不支持团队、组织、角色权限。

## 10. 风险与约束

### 10.1 模型输出不稳定

应通过结构化输出、重试机制、输出校验、提示词版本管理降低风险。

### 10.2 文档解析质量不稳定

PDF 解析可能出现乱码、表格错乱，需要在 MVP 中明确仅保证文本型 PDF 的可用性。

### 10.3 知识库问答幻觉

回答必须优先基于检索内容，无法找到依据时应提示「当前知识库中没有找到足够依据」。

### 10.4 API 成本

系统需要记录每次模型调用的 token 用量、模型名称和调用耗时。

## 11. 验收标准

1. 上传 10 页以内 PDF 后，系统可在 30 秒内完成解析和索引。
2. 用户对已上传文档提问，系统能返回答案和至少 1 条引用来源。
3. 当问题与知识库无关时，系统应明确提示缺少依据。
4. AI 生成任务时，输出必须包含标题、描述、优先级、状态。
5. 用户可以对任务进行新增、编辑、完成、删除。
6. 模型 API Key 未配置时，系统需要给出明确提示。
7. 模型调用失败时，页面不能崩溃，需要展示错误原因和重试入口。
