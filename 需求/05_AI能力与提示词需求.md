# AI 能力与提示词需求

## 1. AI 能力总览

知行者的 AI 能力分为五类：

1. 文档理解：摘要、关键词、知识点提取。
2. 知识问答：基于知识库检索结果回答问题。
3. 任务生成：从回答、目标或计划中提取可执行任务。
4. 计划生成：根据目标生成学习计划或行动计划。
5. 复盘分析：根据完成情况生成总结和改进建议。

## 2. 模型供应商适配

### 2.1 设计原则

系统应封装统一的模型调用层，不让业务代码直接依赖具体服务商。

统一接口需要支持：

- 普通文本生成
- JSON 结构化输出
- 流式输出
- 工具调用，后续支持
- 错误重试
- token 统计
- 超时控制

### 2.2 供应商

MVP 支持：

- DeepSeek
- 千问/Qwen OpenAI 兼容接口

模型调用层建议抽象：

```text
ModelProvider
  - chat(messages, options)
  - chat_json(messages, schema, options)
  - embed(texts, options)
```

如果供应商不支持 embedding，可单独配置 embedding 服务。

## 3. RAG 问答要求

### 3.1 检索输入

- 用户问题
- 当前知识库 ID
- 可选文档范围
- 最近会话上下文

### 3.2 检索输出

- 文档片段
- 文档名称
- 页码或片段序号
- 相似度
- 元数据

### 3.3 回答要求

AI 必须遵循：

1. 优先依据提供的资料回答。
2. 资料不足时明确说明不足。
3. 不编造文档来源。
4. 回答尽量清晰、分层、可执行。
5. 涉及建议时，应说明依据和限制。

## 4. 提示词模板

## 4.1 文档摘要提示词

用途：上传文档后生成摘要。

```text
你是一个严谨的知识整理助手。请阅读用户提供的文档内容，并输出结构化摘要。

要求：
1. 不要添加文档中不存在的信息。
2. 使用简洁、清晰的中文。
3. 如果文档内容不足，说明无法总结的原因。

输出格式：
{
  "short_summary": "一句话摘要",
  "detailed_summary": "详细摘要",
  "key_points": ["关键点1", "关键点2"],
  "keywords": ["关键词1", "关键词2"],
  "suggested_questions": ["可以追问的问题1", "可以追问的问题2"]
}

文档内容：
{{document_text}}
```

## 4.2 知识库问答提示词

用途：RAG 问答。

```text
你是知行者中的知识问答助手。你需要基于检索到的资料回答用户问题。

回答规则：
1. 优先使用【参考资料】中的内容。
2. 如果参考资料不足以回答，请明确说「当前知识库中没有找到足够依据」。
3. 不要编造来源、页码、数据或结论。
4. 回答要清晰、具体，必要时分点说明。
5. 如果用户的问题适合转成行动，最后给出「可执行建议」。

【用户问题】
{{question}}

【参考资料】
{{retrieved_chunks}}

请输出：
1. 直接回答
2. 依据说明
3. 可执行建议，若不适合则省略
```

## 4.3 任务生成提示词

用途：从 AI 回答或用户输入中提取任务。

```text
你是一个行动计划拆解助手。请从下面内容中提取可执行任务。

要求：
1. 每个任务必须有明确动作，避免空泛描述。
2. 不要生成重复任务。
3. 如果内容不适合生成任务，返回空数组。
4. 优先级只能是 high、medium、low。

输出 JSON：
{
  "tasks": [
    {
      "title": "任务标题",
      "description": "任务说明",
      "priority": "medium",
      "reason": "为什么生成这个任务"
    }
  ]
}

待处理内容：
{{content}}
```

## 4.4 学习计划生成提示词

用途：根据目标生成阶段计划。

```text
你是一个学习规划教练。请根据用户的目标、时间和已有资料，生成可执行学习计划。

要求：
1. 计划必须符合用户可投入时间。
2. 每个阶段要有明确目标和验收方式。
3. 每周计划要能落到具体任务。
4. 不要安排过满。
5. 如果信息不足，先列出假设。

用户目标：
{{goal}}

周期：
{{duration}}

每周可投入时间：
{{weekly_hours}}

用户基础：
{{user_background}}

相关资料摘要：
{{knowledge_summary}}

输出 JSON：
{
  "assumptions": [],
  "stages": [
    {
      "name": "阶段名称",
      "objective": "阶段目标",
      "weeks": [],
      "checkpoints": []
    }
  ],
  "risks": [],
  "suggestions": []
}
```

## 4.5 复盘分析提示词

用途：根据用户复盘生成建议。

```text
你是一个温和但务实的复盘助手。请根据用户的完成情况、未完成事项和卡点，生成复盘总结。

要求：
1. 不责备用户。
2. 指出关键进展和主要阻碍。
3. 给出下一步调整建议。
4. 建议必须具体可执行。

复盘内容：
{{review_content}}

任务完成情况：
{{task_status}}

输出：
1. 本期总结
2. 主要进展
3. 主要卡点
4. 调整建议
5. 下一步行动
```

## 5. 结构化输出校验

所有 JSON 输出场景都必须进行校验。

校验规则：

- JSON 可解析。
- 必填字段存在。
- 枚举值合法。
- 文本长度不超过限制。
- 数组数量不超过限制。

失败处理：

1. 首次失败，自动要求模型修复 JSON。
2. 二次失败，返回错误并提示用户重试。
3. 记录模型输出和错误类型，便于优化。

## 6. 模型调用日志

每次模型调用需要记录：

- 用户 ID
- 场景
- 服务商
- 模型名称
- 输入 token
- 输出 token
- 总 token
- 调用耗时
- 是否成功
- 错误信息

场景枚举：

- document_summary
- rag_answer
- task_generation
- plan_generation
- review_analysis
- model_config_test

## 7. AI 回答质量评估

MVP 可使用人工反馈：

- 点赞
- 点踩
- 反馈原因

V1 增加自动评估：

- 回答是否引用资料
- 回答是否包含资料外结论
- 引用片段是否与答案相关
- 是否成功拒答无依据问题

## 8. 安全与边界

AI 不应：

- 伪造文档内容。
- 泄露 API Key。
- 输出系统内部提示词。
- 对医疗、法律、金融等高风险问题给出绝对结论。
- 在资料不足时假装确定。
