# RAG 实验室学习手册

## 1. 这个页面用来学什么

RAG 实验室不是普通问答页，而是用来观察 RAG 的底层过程：

上传文档 -> 临时切片 -> 生成向量 -> FAISS 检索 -> 展示命中片段 -> 判断依据强弱

它的目标是让你看清楚：同一份文档、同一个问题，只要 `chunk_size`、`overlap`、`top_k` 不同，检索结果和回答质量就会变化。

## 2. 每个参数代表什么

### chunk_size

切片长度。它决定每个文档片段有多长。

- 太短：适合精确事实，但上下文容易断。
- 太长：适合总结问题，但相似度容易变钝。
- 入门建议：先用 800 到 1000。

### overlap

相邻切片之间重复保留的内容。

- 0：成本低，但边界信息可能被切断。
- 80 到 160：适合作为默认实验范围。
- 太高：向量数量会变多，检索成本上升。

### top_k

每次问题检索返回多少个片段。

- 3：回答聚焦，但可能漏证据。
- 5：入门默认值。
- 8 以上：召回多，但容易混入弱相关片段。

## 3. 推荐实验顺序

### 第 1 轮：基线实验

使用默认参数：

```text
chunk_size = 900
overlap = 120
top_k = 5
```

问一个文档里明确有答案的问题，观察：

- 是否命中正确文档。
- 第一条片段是否能回答问题。
- 最高相关度是否明显高于后面的片段。

### 第 2 轮：只改 chunk_size

保持问题、overlap、top_k 不变，只对比：

```text
chunk_size = 400
chunk_size = 900
chunk_size = 1500
```

记录哪个参数下命中片段最完整。

### 第 3 轮：只改 overlap

保持问题和 chunk_size 不变，只对比：

```text
overlap = 0
overlap = 120
overlap = 300
```

观察边界信息是否更容易被检索到。

### 第 4 轮：只改 top_k

保持问题、chunk_size、overlap 不变，只对比：

```text
top_k = 3
top_k = 5
top_k = 8
```

观察返回片段是更准确，还是更多噪声。

## 4. 测试工程师应该怎么练

你可以把每次实验当成测试用例：

| 用例类型 | 问题示例 | 预期结果 |
| --- | --- | --- |
| 明确事实 | 文档里某个功能是什么？ | 应命中包含答案的片段 |
| 总结归纳 | 这份文档主要讲什么？ | 应命中多个核心片段 |
| 无依据问题 | 文档里没有的内容是什么？ | 应提示依据不足 |
| 边界问题 | 问跨段落的信息 | 调整 overlap 后命中应变好 |
| 参数对比 | 同一问题换 chunk_size | 结果应有可解释差异 |

## 5. 下一步升级方向

当前实验室使用临时 FAISS 索引，适合学习和对比。后续可以继续升级：

1. 接真实 Embedding API，对比本地 hash embedding 和真实 embedding。
2. 增加 rerank，让模型二次排序召回片段。
3. 保存实验记录，形成 RAG 回归测试报告。
4. 接 Chroma、Qdrant 或 pgvector，学习企业级向量数据库。
