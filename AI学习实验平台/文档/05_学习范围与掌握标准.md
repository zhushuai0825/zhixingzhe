# 学习范围与掌握标准

这份文档回答一个问题：现在学这些够不够，应该学到什么程度才算真的会。

## 1. 当前先学什么

现在先学大模型和算法，不学后端、数据库、部署。

原因很简单：你现在最需要建立的是“AI 应用如何思考和判断”的能力，而不是先陷进工程配置。真正面试或做项目时，最容易拉开差距的不是会不会启动一个服务，而是你能不能讲清：

- 文档为什么要切片。
- Embedding 为什么能做语义匹配。
- 向量库为什么要保存 text、embedding、metadata。
- Rerank 为什么能提升证据质量。
- RAG 为什么必须评测。
- Graph RAG 解决普通 RAG 的什么问题。
- Agent 为什么需要工具、状态、观察和停止条件。
- 推荐算法为什么不是“猜你喜欢”四个字，而是有召回、排序、解释和评估。

## 2. 必学主线

### 2.1 RAG 基础链路

学习顺序：

```text
文本切分 -> Embedding -> 向量库 -> Rerank -> RAG 评测 -> Graph RAG -> Agent -> 模型调用
```

你要能讲清：

- 文档进入系统后，先被解析成文本。
- 长文本不能直接检索，所以要切成 chunk。
- 每个 chunk 会被 Embedding 模型变成向量。
- 向量和原文、来源、页码等 metadata 一起存入向量库。
- 用户问题也会被变成同一空间里的 query 向量。
- 向量库用相似度找 Top K。
- Top K 只是候选证据，不等于最终正确答案。
- Rerank 会重新判断“问题和证据是否真的匹配”。
- 最终回答必须基于证据，不足时要拒答。

动手标准：

- 上传或输入一段文档，能看懂它被切成哪些 chunk。
- 能解释 chunk size 变大或变小会发生什么。
- 能查看 query 向量、chunk 向量和相似度。
- 能解释为什么命中第一条，而不是另一条。
- 能调 Top K、overlap、Rerank 策略，并观察结果变化。
- 能判断一次 RAG 回答是不是证据不足。

### 2.2 Agent 主线

Agent 不是更高级的聊天框。它的核心是：围绕目标，自己决定下一步调用什么工具，并保留轨迹。

你要能讲清：

- LLM 负责推理和决策。
- Tool 负责做具体事情，比如检索知识库、查图谱、重排证据。
- State 保存当前目标、已调用工具、观察结果和中间结论。
- Planner 把目标拆成步骤。
- Observer 读取工具返回。
- Evaluator 判断证据是否足够。
- Reflector 检查是否跑偏、是否要重试或停止。

动手标准：

- 输入一个目标，能看到计划、工具调用、观察、评估、最终结果。
- 关闭 RAG 工具后，能看出 Agent 为什么证据变弱。
- 调低最大步数后，能解释为什么 Agent 还没完成就停了。
- 新增工具后，能看懂工具输入输出怎么进入轨迹。

### 2.3 模型调用与 Prompt 工程

模型调用不是只会发一个 API 请求。你要关心输入、输出、格式、失败和成本。

你要能讲清：

- system prompt 决定角色和边界。
- user prompt 表达本轮任务。
- context 是检索或外部工具给模型的材料。
- JSON Schema 用来限制输出结构。
- temperature 越高，输出越发散。
- token 越多，成本和延迟越高。
- 真实系统必须校验输出，不合格要重试或降级。

动手标准：

- 改 system prompt 后，能观察输出风格变化。
- 删除 JSON 约束后，能看到结构化输出变差。
- 清空上下文后，能判断模型是否开始编。
- 调高 temperature 后，能解释稳定性为什么下降。

## 3. 推荐算法主线

推荐算法和 RAG 都在做“从很多候选里挑出最相关的东西”，所以很值得一起学。

推荐学习顺序：

```text
标签推荐 -> ItemCF -> UserCF -> SVD -> 推荐系统总览
```

### 3.1 标签推荐

它最像人工规则：用户喜欢什么标签，就推荐相同或相近标签的物品。

你要能讲清：

- 用户画像如何由喜欢行为统计出来。
- 物品标签如何和用户画像匹配。
- 冷启动为什么常用标签推荐。
- 标签推荐为什么简单但上限有限。

### 3.2 ItemCF

ItemCF 看的是物品之间的共现：喜欢 A 的人也喜欢 B，说明 A 和 B 可能相似。

你要能讲清：

- 用户-物品行为矩阵是什么。
- 共现矩阵怎么来。
- 为什么要做余弦归一化，避免热门物品把结果带偏。
- 推荐分为什么来自“已喜欢物品对候选物品的相似度累加”。
- 为什么已喜欢物品要过滤掉。

### 3.3 UserCF

UserCF 看的是人和人像不像：和你相似的人喜欢什么，可以推荐给你。

你要能讲清：

- 用户相似度如何由共同喜欢计算。
- 相似用户如何给候选物品贡献分数。
- UserCF 为什么容易受用户兴趣变化影响。
- UserCF 和 ItemCF 的区别是什么。

### 3.4 SVD

SVD 把稀疏行为矩阵压缩成用户隐向量和物品隐向量。

你要能讲清：

- 为什么真实推荐矩阵很稀疏。
- 隐向量不是人工标签，而是模型学出来的潜在兴趣维度。
- 用户向量和物品向量点积越大，预测越喜欢。
- SVD 为什么比简单共现更能泛化，但解释性更弱。

## 4. 暂时不学什么

这些不是没用，而是现在不放在主线：

- FastAPI。
- PostgreSQL。
- Docker。
- Nginx。
- 云服务器。
- 权限、备份、上线运维。

等你能把 RAG、Agent、推荐算法讲清楚并用实验台跑明白，再学这些会更轻松。

## 5. 你学会后的判断标准

如果你能做到下面这些，就说明当前阶段够了：

- 能用白话讲清一个 RAG 系统从文档到回答的全流程。
- 能指出 RAG 回答错了是切片、召回、重排、证据不足还是 Prompt 的问题。
- 能解释 Embedding、相似度、向量库、Rerank 各自负责什么。
- 能用 Agent 轨迹解释一次目标是怎么变成工具调用和最终结果的。
- 能对比标签推荐、ItemCF、UserCF、SVD 的输入、算法思想、优缺点和适用场景。
- 能用平台自己新增数据，观察结果变化，而不是只看固定演示。

## 6. 面试表达模板

### 问：RAG 是什么？

可以这样答：

RAG 是检索增强生成。它不是把文档直接丢给大模型，而是先把文档解析、切片、向量化并存入向量库。用户提问时，系统把问题也向量化，在同一个向量空间里找相似 chunk，再把这些 chunk 作为上下文给大模型回答。这样回答可以基于企业自己的资料，并且能展示引用来源。

继续补一句会更像工程经验：

但向量库命中不等于答案一定正确，所以工业系统通常还会加 Rerank、证据门、引用校验和评测集。

### 问：Agent 和普通 RAG 问答有什么区别？

可以这样答：

普通 RAG 通常是一次检索一次回答。Agent 更像一个有工具的执行流程，它会先理解目标，再决定要不要检索、调用哪个工具、证据够不够、是否需要重试，最后生成结果。Agent 的重点不是“模型更聪明”，而是把计划、工具、观察、评估和停止条件组织起来，让复杂任务更可控。

### 问：ItemCF 和 UserCF 有什么区别？

可以这样答：

ItemCF 计算物品和物品的相似度，核心是假设“用户喜欢过的物品，可以推荐相似物品”。UserCF 计算用户和用户的相似度，核心是假设“和我兴趣相似的人喜欢的东西，我也可能喜欢”。ItemCF 通常更稳定，适合物品变化不太快的场景；UserCF 更依赖用户群体相似，但用户兴趣变化时波动会更明显。

### 问：为什么需要 Rerank？

可以这样答：

向量召回适合快速从大量 chunk 里找语义相近的候选，但它可能把“词义接近但不能回答问题”的内容排在前面。Rerank 会拿 query 和候选 chunk 做更精细的相关性判断，把真正能支撑答案的证据排到前面。企业 RAG 里 Rerank 很常见，因为它能明显提升证据质量和回答可靠性。
