# 文本切分零基础学习手册

## 1. 文本切分是什么

文本切分就是把一篇长文档拆成多个小片段。

这些小片段通常叫：

```text
chunk
```

在 RAG 里，chunk 是后续检索的基本单位。

也就是说：

```text
不是整篇文档一起进入向量数据库，
而是一个个 chunk 进入向量数据库。
```

## 2. 为什么不能整篇文档直接 embedding

原因有几个：

### 2.1 文档太长

Embedding 模型通常有输入长度限制。

一整份 PDF 或 Word 可能远远超过限制。

### 2.2 检索不精准

如果整篇文档只有一个向量，用户问一个很细的问题时，系统只能召回整篇文档。

这会导致：

- 上下文太长。
- 噪声太多。
- 大模型不知道该看哪里。

### 2.3 成本高

大模型 prompt 越长，成本越高，速度越慢。

好的 RAG 应该只给模型最相关的证据片段。

## 3. chunk 在 RAG 里的位置

完整流程：

```text
文档解析
  -> 文本切分
  -> chunk embedding
  -> 写入向量数据库
  -> 用户问题 embedding
  -> 向量检索 Top K chunk
  -> 大模型基于 chunk 回答
```

所以文本切分是 RAG 的第一道质量关。

切分不好，后面 embedding、向量库、rerank 都会受影响。

## 4. 什么是 chunk size

chunk size 控制每个 chunk 的最大长度。

例如：

```text
chunk size = 300
```

意思是每个 chunk 尽量不超过 300 个字符。

当前实验室采用更容易理解的口径：`chunk size` 表示一个 chunk 的总长度上限。也就是说，如果某个 chunk 有 overlap，overlap 也算在这个总长度里。

真实工程里，也可能按 token 数算，而不是字符数。

## 5. chunk 太短会怎样

如果 chunk 太短：

- 语义不完整。
- 前因后果断开。
- 一个答案可能分散在多个 chunk 里。
- 大模型拿到片段后看不懂背景。

例子：

```text
“它可以提升准确率。”
```

这个片段太短，用户不知道“它”是什么。

## 6. chunk 太长会怎样

如果 chunk 太长：

- 一个 chunk 里混入多个主题。
- 向量表示变得模糊。
- 检索命中后带入很多无关内容。
- prompt 成本更高。

例子：

一个 chunk 同时讲：

- 文本切分。
- 向量数据库。
- Agent。
- 部署。

那它的向量会混合多个主题，检索时不够精准。

## 7. 什么是 overlap

overlap 是相邻 chunk 之间重复保留的内容。

例如：

```text
chunk size = 100
overlap = 20
```

第一个 chunk 覆盖：

```text
0-100
```

第二个 chunk 可能覆盖：

```text
80-180
```

中间 `80-100` 就是重叠部分。

换句话说，`chunk size = 100`、`overlap = 20` 时，第二个 chunk 不是 120 个字符，而是大约：

```text
前一个 chunk 的尾部 20 字 + 新内容 80 字
```

所以每个 chunk 的总长度仍然尽量不超过 100。

## 8. 为什么需要 overlap

因为答案可能刚好出现在切片边界。

如果没有 overlap，句子可能被切断。

overlap 可以让边界附近的上下文在两个 chunk 里都保留一份。

## 9. overlap 太大有什么问题

overlap 不是越大越好。

太大会导致：

- 重复内容太多。
- chunk 数量变多。
- embedding 成本变高。
- 向量库体积变大。
- 检索结果重复。

所以 overlap 要适中。

## 10. 常见切分策略

### 10.1 字符切分

按固定字符数切。

优点：

- 最简单。
- 容易实现。

缺点：

- 容易切断句子。
- 容易破坏语义。

### 10.2 段落切分

优先按空行和段落切。

优点：

- 更符合人类阅读结构。
- 语义更完整。

缺点：

- 段落长度不稳定。
- 遇到超长段落仍然需要硬切。

### 10.3 递归切分

递归切分会按优先级尝试：

```text
标题 -> 段落 -> 句子 -> 标点 -> 空格 -> 字符
```

它尽量保持自然边界，是 RAG 工程里很常见的思路。

## 11. 怎么判断切分好不好

可以看：

- chunk 是否语义完整。
- chunk 长度是否稳定。
- 是否大量切断句子。
- 是否有过多重复。
- 用户问题能否命中正确 chunk。
- 返回给大模型的上下文是否干净。

## 12. 一句话总结

```text
文本切分决定了知识库被检索时的最小证据单元。

切得好，RAG 更容易命中正确证据；
切得差，后面模型再强也可能找不到答案。
```
