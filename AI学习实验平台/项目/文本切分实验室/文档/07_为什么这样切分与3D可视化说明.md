# 为什么这样切分与 3D 可视化说明

## 1. 为什么不是简单按字数切

RAG 里最重要的不是“每片刚好一样长”，而是“每片尽量语义完整”。

如果只按字数切，可能出现：

```text
前半句在 chunk-1，后半句在 chunk-2。
标题在 chunk-1，解释在 chunk-2。
问题答案刚好被切到边界两侧。
```

这样做 embedding 时，每个 chunk 的语义会变弱，用户提问时也不容易命中正确证据。

## 2. 递归切分为什么先看自然边界

递归切分会按这个优先级尝试：

```text
标题
  -> 段落
  -> 句子
  -> 换行
  -> 逗号
  -> 空格
  -> 字符硬切
```

它的目标是：

- 能按标题切，就不要按句子硬切。
- 能保留段落完整，就不要打断段落。
- 实在太长，才退回更小单位。

所以你调 Chunk Size 时，切片数量有时不会马上变化。原因是当前参数还没有跨过自然边界阈值。

## 3. Chunk Size 和 Overlap 怎么共同工作

本实验室使用这个口径：

```text
chunk 总长度 <= chunk size
```

如果：

```text
chunk size = 260
overlap = 60
```

那么后续 chunk 通常是：

```text
上一个 chunk 尾部 60 字 + 新内容 200 字
```

这样 overlap 不会额外把 chunk 撑大。

## 4. 3D 画布怎么看

3D 画布里的每个块都是一个 chunk。

- 从左到右：原文位置从前到后。
- 块越宽：这个 chunk 覆盖的原文越多。
- 深色厚片：overlap，即从上一片复制过来的尾部内容。
- 橙色边框：可能有边界风险或过短风险。
- 拖拽：旋转观察切片关系。
- 滚轮：放大缩小。
- 悬停：查看这个 chunk 为什么这样切。

## 5. 切完之后进入哪里

切分只是 RAG 的第一步。

后续链路是：

```text
chunk
  -> embedding
  -> 向量库 collection
  -> 查询问题 query embedding
  -> 相似度 Top K
  -> Rerank
  -> 大模型基于引用回答
```

向量库里通常保存：

```text
id
text
embedding vector
metadata
```

metadata 包含来源文件、页码、章节等信息。它不是用来生成答案的，而是用来过滤、引用和管理数据的。
