# Chunk Size 与 Overlap 详解

## 1. 两个参数的核心作用

```text
chunk size：控制每片多大。
overlap：控制相邻片重复多少。
```

本实验室的实现口径是：overlap 包含在 chunk size 里面。也就是说，`chunk size = 500`、`overlap = 80` 时，新 chunk 不是 580 字，而是大约 80 字来自上片尾部、420 字来自新内容。

它们共同决定：

- chunk 数量。
- 每个 chunk 的语义完整度。
- embedding 成本。
- 向量库体积。
- 检索命中质量。

## 2. chunk size 怎么选

没有永远正确的值。

要看：

- 文档类型。
- 模型上下文长度。
- embedding 模型输入限制。
- 问题粒度。
- 业务是否需要引用准确。

常见经验：

```text
FAQ：短一些。
技术文档：中等。
法律/合同：需要更谨慎，最好按条款结构切。
论文/报告：按标题层级和段落切。
```

## 3. overlap 怎么选

overlap 一般是 chunk size 的一小部分。

例如：

```text
chunk size = 500
overlap = 50 到 100
```

它的目的不是制造大量重复，而是保护边界上下文。

如果你把 overlap 调大，chunk 数量可能会增加，因为每次前进的新内容变少了；但在递归切分里，系统会优先保留标题、段落、句子等自然边界，所以某些参数区间内切片数量可能暂时不变。

## 4. 参数组合示例

### 4.1 小 chunk + 小 overlap

适合：

- FAQ。
- 问题答案短。
- 结构清晰。

风险：

- 上下文不足。

### 4.2 中 chunk + 中 overlap

适合：

- 产品文档。
- 技术说明。
- 测试方案。

这是很多 RAG 项目的起步配置。

### 4.3 大 chunk + 大 overlap

适合：

- 需要长上下文的复杂文档。

风险：

- 成本高。
- 噪声多。
- 检索结果重复。

## 5. 为什么要看边界风险

如果 chunk 结尾不是句号、换行、冒号等自然边界，可能说明它被硬切断。

这种 chunk 可能有边界风险。

边界风险不一定代表错误，但值得检查。

## 6. 为什么真实系统按 token 切

当前实验室为了入门，用字符长度。

真实系统更常按 token 切。

原因是模型限制通常是 token 数，不是字符数。

例如：

```text
embedding 模型最大输入 8192 tokens
```

这时 chunk size 应该按 token 控制。

## 7. 和向量库的关系

chunk 越多：

- 需要计算的 embedding 越多。
- 向量库记录越多。
- 检索成本越高。

chunk 越少：

- 每个 chunk 更长。
- 语义更混杂。
- 检索可能不精准。

所以切分是效果和成本之间的平衡。
