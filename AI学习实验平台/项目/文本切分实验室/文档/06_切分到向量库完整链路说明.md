# 从文本切分到向量库的完整链路说明

## 1. 文本切分在 RAG 里的位置

文本切分不是孤立步骤。

它前面通常是：

```text
PDF/Word/网页
  -> 文档解析
  -> 提取纯文本
```

它后面通常是：

```text
chunk
  -> embedding
  -> 写入向量数据库
  -> 查询召回
```

所以切分质量会直接影响后面的 embedding 和检索。

## 2. 一个 chunk 后面会发生什么

每个 chunk 会变成一条向量库记录。

通常包含：

```text
id：chunk 唯一编号
text：chunk 原文
embedding：chunk 向量
metadata：来源文件、页码、标题、位置等信息
```

真实 RAG 查询时，系统返回的不只是向量分数，还要返回 chunk 原文和 metadata。

这样大模型才能引用来源。

## 3. chunk size 如何影响向量库

chunk size 越小：

- chunk 数量越多。
- embedding 调用次数越多。
- 向量库记录越多。
- 检索更细，但上下文可能不足。

chunk size 越大：

- chunk 数量越少。
- 向量库记录更少。
- 上下文更多，但噪声也更多。

## 4. overlap 如何影响向量库

overlap 会制造重复内容。

适当 overlap 有好处：

- 保护边界上下文。
- 防止答案刚好被切断。

但 overlap 太大会导致：

- 存储重复。
- 召回结果重复。
- 成本变高。

## 5. 后续向量库实验室会学什么

下一步向量库实验室会把这条链路继续往下接：

```text
chunk
  -> embedding
  -> collection
  -> upsert
  -> query
  -> metadata filter
  -> delete
```

你会看到：

- chunk 存在哪里。
- embedding 怎么存。
- 查询时怎么命中。
- 命中结果为什么带原文和来源。
- 删除文档时为什么要删除对应 chunk。
