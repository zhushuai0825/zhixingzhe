# 技术架构规划

## 1. 总体架构

```text
前端输入文本和参数
  -> FastAPI /api/split
  -> 后端执行切分策略
  -> 返回 chunks 和质量统计
  -> 前端展示列表和 Canvas 边界图
```

## 2. 前端

技术：

- HTML。
- CSS。
- JavaScript。
- Canvas。

职责：

- 输入长文本。
- 调整 chunk size 和 overlap。
- 切换切分策略。
- 展示切片列表。
- 绘制切片边界画布。
- 后端不可用时提供本地兜底切分。

## 3. 后端

技术：

- Python。
- FastAPI。
- Pydantic。

职责：

- 实现字符切分。
- 实现段落切分。
- 实现递归切分。
- 计算切片质量统计。
- 返回完整切片数据。

## 4. 当前不使用数据库

文本切分是预处理阶段。

当前项目目标是学习算法和参数，不是持久化数据。

后续向量库实验室会再学习：

- chunk 如何写入数据库。
- embedding 如何存储。
- metadata 如何保存。
- 如何删除和重建索引。

## 5. 切分策略

### 5.1 字符切分

固定长度切分。

优点：

- 简单。
- 可控。
- 好理解。

缺点：

- 容易切断句子。
- 容易破坏语义。

### 5.2 段落切分

优先保留段落。

优点：

- 语义更完整。

缺点：

- 段落长度不稳定。
- 超长段落仍然需要继续拆。

### 5.3 递归切分

按分隔符优先级逐步尝试：

```text
标题 -> 段落 -> 句号 -> 分号 -> 换行 -> 逗号 -> 空格 -> 字符
```

优点：

- 更接近实际 RAG 工程。
- 尽量保持自然边界。

缺点：

- 实现更复杂。
- 参数仍然需要调试。
