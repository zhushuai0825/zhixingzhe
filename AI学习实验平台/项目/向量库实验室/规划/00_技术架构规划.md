# 技术架构规划

## 1. 当前架构

```text
浏览器前端
  ├─ 表单：Upsert / Query / Delete / Reset
  ├─ Collection 列表
  ├─ 向量维度预览
  └─ 3D 命中空间

FastAPI 后端 8050
  ├─ 内存 store
  ├─ 教学 embedding
  ├─ 相似度计算
  └─ REST API
```

## 2. 为什么先用内存向量库

直接上 Chroma、Milvus 或 pgvector 会把学习重点分散到安装、版本、驱动和部署问题上。

当前阶段最重要的是看懂概念：

- 存什么。
- 怎么存。
- 查什么。
- 怎么排序。
- 为什么需要 metadata。

所以先用一个很小的内存 store 模拟向量库行为。

## 3. 数据模型

一条向量记录：

```json
{
  "id": "chunk-1",
  "text": "向量数据库负责存储 chunk embedding...",
  "metadata": {
    "source": "vector_db.md",
    "section": "向量库",
    "page": "2"
  },
  "embedding": {
    "tokens": ["向", "量", "数", "据", "库"],
    "vector": [0, 0.5, 0.5, 0, 0, 0, 0, 0, 0, 0, 0, 0]
  }
}
```

字段含义：

- `id`：删除、覆盖、引用都依赖它。
- `text`：最终给大模型的证据原文。
- `metadata`：来源、章节、页码、权限、类型。
- `embedding`：用于相似度检索的向量。

## 4. API 设计

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| GET | `/health` | 健康检查 |
| GET | `/api/state` | 获取 collection 状态 |
| POST | `/api/reset` | 重置示例数据 |
| POST | `/api/upsert` | 写入或覆盖记录 |
| POST | `/api/query` | 查询 Top K |
| DELETE | `/api/documents/{id}` | 删除记录 |

## 5. 后续真实架构

教学版看懂后，真实工程可以升级为：

```text
文档上传
  ↓
解析 PDF / Word / HTML
  ↓
文本切分
  ↓
Embedding 模型
  ↓
向量库
  ├─ Chroma：入门、本地快
  ├─ FAISS：本地高性能索引
  ├─ Milvus：企业级向量数据库
  └─ PostgreSQL + pgvector：业务数据和向量一起存
  ↓
Top K 检索
  ↓
Rerank
  ↓
大模型基于引用回答
```

## 6. 成功标准

- 学习者能画出向量库在 RAG 中的位置。
- 学习者能解释 collection 和 metadata。
- 学习者能用 API 写入、查询、删除数据。
- 学习者能理解为什么相似度高的记录排在前面。
