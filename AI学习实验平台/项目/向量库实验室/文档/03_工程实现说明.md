# 工程实现说明

## 1. 前端文件

```text
前端原型/index.html
前端原型/styles.css
前端原型/app.js
```

前端负责：

- 项目切换菜单。
- 入库表单。
- 查询表单。
- collection 列表。
- 命中结果。
- 向量维度条形图。
- 3D 命中空间。

## 2. 后端文件

```text
后端/app/main.py
后端/requirements.txt
```

后端负责：

- 保存内存 store。
- 生成教学 embedding。
- 计算相似度。
- 提供 REST API。

当前 store 是内存数据：页面刷新不会清空，因为数据在 8050 后端进程里；但后端重启后会恢复到 5 条示例数据。如果想手动清理测试数据，可以点击页面里的 `重置示例数据`。

## 3. 为什么前端也有兜底算法

如果 FastAPI 后端没启动，页面仍然可以用浏览器本地算法运行。

这样做是为了学习体验稳定：

- 后端启动时：使用真实 API 返回的数据。
- 后端未启动时：使用前端内置教学数据，不会白屏。

页面顶部会显示当前状态：

- `FastAPI · 8050 在线`
- `离线教学向量库`

## 4. 教学 Embedding 如何工作

当前版本不是大模型 embedding，而是教学用的语义维度 embedding。

它把文本映射到 12 个维度：

- RAG 检索。
- Embedding 向量化。
- 向量数据库。
- 相似度计算。
- 文档切分。
- Rerank 重排。
- Agent 工具。
- 推荐算法。
- 生成回答。
- 部署工程。
- 评测质量。
- 学习概念。

这样你可以清楚看到“为什么命中”，而不是只看到几百或几千个难以理解的数字。

## 5. 相似度计算

后端会对向量做归一化，然后计算点积：

```text
score = query_vector · chunk_vector
```

归一化之后，点积可以理解成余弦相似度：

- 越接近 1，语义方向越接近。
- 越接近 0，语义关系越弱。

## 6. API 示例

### 6.1 查询状态

```bash
curl http://127.0.0.1:8050/api/state
```

### 6.2 写入记录

```bash
curl -X POST http://127.0.0.1:8050/api/upsert \
  -H 'Content-Type: application/json' \
  -d '{
    "id": "chunk-new",
    "text": "向量库保存 chunk embedding 和 metadata。",
    "metadata": {
      "source": "custom.md",
      "section": "练习"
    }
  }'
```

### 6.3 查询 Top K

```bash
curl -X POST http://127.0.0.1:8050/api/query \
  -H 'Content-Type: application/json' \
  -d '{
    "query": "embedding 和向量数据库是什么关系？",
    "topK": 3,
    "source": ""
  }'
```

### 6.4 删除记录

```bash
curl -X DELETE http://127.0.0.1:8050/api/documents/chunk-new
```

## 7. 迁移到真实向量库时改哪里

需要替换的是后端 store 层：

当前：

```text
dict[str, record]
```

未来：

```text
Chroma collection
FAISS index + metadata store
Milvus collection
PostgreSQL table + pgvector
```

前端可以尽量不变，因为前端只关心 API 返回的结构。
