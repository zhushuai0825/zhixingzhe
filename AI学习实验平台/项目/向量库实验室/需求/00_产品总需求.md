# 向量库实验室产品需求

## 1. 背景

在 RAG 系统里，文本切分和 Embedding 只是前半段。真正要让系统可查询，还必须把每个 chunk 的向量、原文和来源信息存起来。

很多初学者会把向量库理解成“大模型数据库”，这是不准确的。向量库主要负责检索，不负责生成回答。它接收 query vector，返回相似的 chunk。

## 2. 目标用户

- 正在学习 RAG 的初学者。
- 想理解向量数据库工程流程的测试工程师。
- 想从“会调 API”进阶到“会设计 RAG 系统”的学习者。

## 3. 学习目标

用户完成本项目后，应能说清楚：

- 向量库和普通数据库有什么不同。
- 一条向量记录为什么要同时保存 `id`、`text`、`embedding`、`metadata`。
- Upsert、Query、Delete 分别解决什么问题。
- Top K 不是大模型猜出来的，而是相似度排序出来的。
- Metadata 为什么对企业知识库很重要。
- 教学版内存向量库如何迁移到真实向量库。

## 4. 功能范围

### 4.1 项目总览

用流程图解释：

```text
Chunk → Embedding → Collection → Query → Top K
```

### 4.2 向量库实验台

必须支持：

- 新增或覆盖一条 chunk 记录。
- 填写文档 ID、文本、source、section。
- 查询当前 collection。
- 设置 Top K。
- 按 source 做 metadata 过滤。
- 查看命中结果、相似度分数和命中原因。
- 查看 collection 中全部记录。
- 删除指定记录。
- 预览选中记录的向量维度。

### 4.3 命中可视化

必须支持：

- 以 3D 空间展示 query vector 和 chunk vector。
- 命中越强的点越靠近 query。
- 支持鼠标拖拽旋转。
- 支持滚轮缩放。
- 悬停节点展示原文、来源和分数。

### 4.4 学习路线

解释向量库学完后如何进入：

- Rerank。
- Hybrid Search。
- pgvector。
- Chroma / Milvus。
- RAG 评测。

## 5. 非目标

当前版本暂不做：

- 真实 PDF 上传解析。
- 真实大模型生成回答。
- 用户登录。
- 多租户权限。
- 生产级持久化。

这些能力会放到知行者 AI 实验室或后续真实工程模块里。

## 6. 验收标准

- 页面可从总平台切换进入。
- 后端启动后，页面能显示 FastAPI 在线。
- 能 upsert 新记录，并在 collection 中看到。
- 能 query 出 Top K 命中结果。
- 能删除记录。
- 能重置示例数据。
- 3D 画布可拖拽、缩放、悬停查看信息。
- 文档能解释清楚向量库在 RAG 中的位置。
