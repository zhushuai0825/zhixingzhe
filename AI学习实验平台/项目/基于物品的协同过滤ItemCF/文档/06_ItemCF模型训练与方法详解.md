# ItemCF 模型训练与方法详解

这份文档专门回答一个问题：

```text
ItemCF 到底是怎么“训练”出来的？
```

先给结论：

```text
ItemCF 的“训练”不是训练神经网络，
不是反向传播，
不是调大模型参数，
也不是把文本变成 embedding。

ItemCF 的训练，本质是从用户行为日志里统计物品之间的共现关系，
再把共现关系归一化成物品相似度矩阵。
```

你可以把它理解成：

```text
用户行为数据
  -> 统计哪些物品经常被同一批用户一起喜欢
  -> 计算物品和物品之间的相似度
  -> 保存或缓存相似度矩阵
  -> 用户来请求推荐时，用他的历史喜欢物品去匹配相似物品
```

这就是当前系统正在做的事。

---

## 1. 推荐系统里的“模型”是什么

很多人一听到“模型”，就会想到大模型、神经网络、训练集、loss、梯度下降。

但推荐系统里，“模型”不一定是神经网络。

在 ItemCF 里，模型可以理解成这几份计算结果：

- 用户-物品行为矩阵。
- 物品共现矩阵。
- 物品相似度矩阵。
- 每个用户的候选物品得分。

其中最核心的是：

```text
物品相似度矩阵
```

例如：

| 物品 | RAG 系统实战 | Chroma 向量数据库 | Agent 工具调用 |
| --- | ---: | ---: | ---: |
| RAG 系统实战 | 1.000 | 0.667 | 0.408 |
| Chroma 向量数据库 | 0.667 | 1.000 | 0.000 |
| Agent 工具调用 | 0.408 | 0.000 | 1.000 |

这个矩阵就是 ItemCF 最重要的模型结果。

它表示：

```text
每两个物品之间有多相似。
```

---

## 2. ItemCF 和大模型训练有什么区别

### 2.1 大模型训练

大模型训练通常是：

```text
大量文本/图像/音频数据
  -> 神经网络
  -> 计算 loss
  -> 反向传播
  -> 更新参数
  -> 得到模型权重
```

特点：

- 需要 GPU。
- 数据量很大。
- 参数很多。
- 训练成本高。
- 训练完成后得到模型权重文件。

### 2.2 ItemCF 训练

ItemCF 训练通常是：

```text
用户行为日志
  -> 构造用户-物品矩阵
  -> 统计物品共现
  -> 计算相似度
  -> 得到相似度矩阵
```

特点：

- 不需要 GPU。
- 不需要反向传播。
- 小数据用普通电脑就能跑。
- 结果非常可解释。
- 训练结果通常是相似度表或相似度矩阵。

所以，ItemCF 更像“统计计算”，不是深度学习训练。

## 3. 当前系统用的是什么方法

当前系统使用的是最经典、最适合入门的 ItemCF 方法：

```text
基于隐式反馈的物品协同过滤
```

拆开解释：

### 3.1 基于物品

系统不是先找“和你相似的人”，而是先找“和你喜欢过的东西相似的东西”。

如果你喜欢：

```text
RAG 系统实战
LangChain 工作流
Agent 工具调用
```

系统会问：

```text
哪些物品经常和这些物品被同一批用户一起喜欢？
```

然后把这些相似物品推荐给你。

### 3.2 协同过滤

“协同”指很多用户共同产生行为。

“过滤”指系统根据这些共同规律筛选出你可能喜欢的物品。

它不需要知道物品内容是什么。

例如，系统不需要读懂“RAG 系统实战”这几个字，它只看行为：

```text
喜欢 RAG 系统实战的人，也经常喜欢 Chroma 向量数据库。
```

于是它认为这两个物品相似。

### 3.3 隐式反馈

当前系统只有“喜欢”行为，没有 1 到 5 分评分。

这种行为叫隐式反馈。

常见隐式反馈包括：

- 点击。
- 浏览。
- 收藏。
- 加购。
- 购买。
- 点赞。
- 完播。
- 下载。

当前项目里，为了学习清楚，只使用：

```text
like = 1
```

没有喜欢记录就当成：

```text
unknown = 0
```

注意，0 不一定代表“不喜欢”，更多时候只是“没有观察到行为”。

## 4. 训练数据长什么样

当前系统有三类表。

### 4.1 用户表 users

保存用户是谁。

```text
id, name, note
```

例子：

```text
u2, 小周, 当前手算案例用户，喜欢科幻电影和太空纪录片。
```

### 4.2 物品表 items

保存可以被推荐的物品。

```text
id, title, category
```

例子：

```text
i8, RAG 系统实战, AI
i14, Chroma 向量数据库, AI
i24, 前端可视化 Canvas, 前端
```

### 4.3 行为表 interactions

保存用户做过什么行为。

```text
user_id, item_id, event_type, weight, created_at
```

当前系统只使用：

```text
event_type = like
weight = 1
```

例子：

```text
u2 喜欢 i8
u2 喜欢 i9
u3 喜欢 i8
u3 喜欢 i14
```

训练 ItemCF 最重要的就是这张行为表。

## 5. 第一步：把行为表变成用户-物品矩阵

行为表是数据库里的事件记录。

算法更容易处理矩阵，所以第一步要把行为变成矩阵。

假设有 3 个用户、4 个物品：

| 用户 | RAG | Chroma | Agent | Docker |
| --- | ---: | ---: | ---: | ---: |
| u1 | 1 | 1 | 0 | 0 |
| u2 | 1 | 0 | 1 | 0 |
| u3 | 0 | 1 | 1 | 1 |

含义：

- 1：用户喜欢过这个物品。
- 0：没有观察到喜欢行为。

这一步在当前后端里对应：

```text
build_user_item_matrix()
```

它从 PostgreSQL 读取 interactions，然后生成每个用户对每个物品的 0/1 向量。

---

## 6. 第二步：统计物品共现

共现就是：

```text
有多少用户同时喜欢了两个物品。
```

继续用上面的矩阵：

| 用户 | RAG | Chroma |
| --- | ---: | ---: |
| u1 | 1 | 1 |
| u2 | 1 | 0 |
| u3 | 0 | 1 |

RAG 和 Chroma 同时为 1 的用户只有 u1。

所以：

```text
co_count(RAG, Chroma) = 1
```

再看 RAG 和 Agent：

| 用户 | RAG | Agent |
| --- | ---: | ---: |
| u1 | 1 | 0 |
| u2 | 1 | 1 |
| u3 | 0 | 1 |

共同喜欢它们的用户只有 u2。

所以：

```text
co_count(RAG, Agent) = 1
```

共现矩阵大概长这样：

| 物品 | RAG | Chroma | Agent | Docker |
| --- | ---: | ---: | ---: | ---: |
| RAG | 2 | 1 | 1 | 0 |
| Chroma | 1 | 2 | 1 | 1 |
| Agent | 1 | 1 | 2 | 1 |
| Docker | 0 | 1 | 1 | 1 |

对角线表示一个物品自己和自己共现，也就是喜欢这个物品的用户数。

这一步在当前后端里对应：

```text
build_item_similarity()
```

函数内部会先构造：

```text
item_user_sets
```

意思是：

```text
每个物品被哪些用户喜欢过。
```

然后用集合交集计算共现：

```text
left_users & right_users
```

---

## 7. 第三步：把共现变成相似度

只看共现次数是不够的。

原因是热门物品天然容易和所有东西共现。

例如：

```text
Python 实战
```

可能很多人都喜欢。它就会和很多物品产生共现。

如果只看共现次数，它会占很大优势，系统会越来越偏向热门物品。

所以当前系统使用余弦归一化公式：

```text
sim(i, j) = co_count(i, j) / sqrt(count(i) * count(j))
```

解释每个符号：

```text
i：物品 i
j：物品 j
co_count(i, j)：同时喜欢 i 和 j 的用户数
count(i)：喜欢 i 的用户数
count(j)：喜欢 j 的用户数
sqrt：平方根
sim(i, j)：物品 i 和物品 j 的相似度
```

### 7.1 为什么要除以 sqrt(count(i) * count(j))

假设：

```text
co_count(A, B) = 2
count(A) = 2
count(B) = 2
```

那么：

```text
sim(A, B) = 2 / sqrt(2 * 2) = 1
```

这表示喜欢 A 的用户基本也喜欢 B，相似度非常强。

再看热门物品：

```text
co_count(A, Hot) = 2
count(A) = 2
count(Hot) = 100
```

那么：

```text
sim(A, Hot) = 2 / sqrt(2 * 100) = 2 / 14.142 = 0.141
```

虽然共现也是 2，但 Hot 太热门了，所以相似度会被压低。

这就是归一化的意义：

```text
不要让热门物品仅仅因为热门就到处高相似。
```

### 7.2 为什么这个公式像余弦相似度

每个物品都可以看成一个用户维度上的 0/1 向量。

例如：

```text
RAG = [1, 1, 0]
Chroma = [1, 0, 1]
```

两个向量点积就是共现次数。

两个向量长度分别是：

```text
sqrt(count(RAG))
sqrt(count(Chroma))
```

所以余弦相似度就是：

```text
点积 / 两个向量长度相乘
```

也就是：

```text
co_count(i, j) / sqrt(count(i) * count(j))
```

## 8. 第四步：生成候选物品

对某个用户做推荐时，不能把他已经喜欢过的物品再推荐给他。

所以候选集是：

```text
候选物品 = 全部物品 - 当前用户已喜欢物品
```

例如当前用户喜欢：

```text
RAG 系统实战
LangChain 工作流
Agent 工具调用
```

系统会把这些排除掉，只给剩余物品打分。

这一步在当前后端里对应：

```text
score_candidates_for_user()
```

里面有一段逻辑：

```text
如果 candidate_id 在 liked_set 里，就 continue
```

意思是跳过已喜欢物品。

---

## 9. 第五步：计算候选物品推荐分

候选物品的分数来自：

```text
它和用户历史喜欢物品的相似度之和。
```

公式：

```text
score(u, j) = sum(sim(j, i) * preference(u, i))
```

解释：

```text
u：用户
j：候选物品
i：用户已经喜欢过的历史物品
sim(j, i)：候选物品 j 和历史物品 i 的相似度
preference(u, i)：用户 u 对历史物品 i 的偏好强度
```

当前项目里，喜欢行为权重都是 1，所以公式可以简化成：

```text
score(u, j) = sim(j, 历史物品1) + sim(j, 历史物品2) + sim(j, 历史物品3) + ...
```

例子：

当前用户喜欢：

```text
RAG 系统实战
LangChain 工作流
Agent 工具调用
```

候选物品是：

```text
Graph RAG 知识图谱
```

如果相似度是：

```text
sim(Graph RAG, RAG 系统实战) = 0.40
sim(Graph RAG, LangChain 工作流) = 0.20
sim(Graph RAG, Agent 工具调用) = 0.35
```

那么：

```text
score(当前用户, Graph RAG) = 0.40 + 0.20 + 0.35 = 0.95
```

这就是推荐分。

---

## 10. 第六步：排序并取 Top N

所有候选物品都有分数以后，系统按分数从高到低排序。

然后取前 N 个：

```text
Top 5
Top 10
Top 20
```

当前前端展示的是推荐列表和推荐解释。

一个推荐结果不仅有总分，还会保留每个历史物品的贡献：

```text
推荐分 = 来源物品 A 的贡献 + 来源物品 B 的贡献 + 来源物品 C 的贡献
```

这就是为什么页面可以解释：

```text
这个推荐是被哪些已喜欢物品推上来的。
```

## 11. 当前系统代码如何对应算法流程

后端文件：

```text
后端/app/main.py
```

核心函数：

| 函数 | 作用 |
| --- | --- |
| `load_users()` | 从 PostgreSQL 读取用户。 |
| `load_items()` | 从 PostgreSQL 读取物品。 |
| `load_interactions()` | 从 PostgreSQL 读取喜欢行为。 |
| `build_user_item_matrix()` | 把行为变成用户-物品 0/1 矩阵。 |
| `build_item_similarity()` | 计算共现矩阵、相似度矩阵、物品被喜欢次数。 |

请求链路：

```text

  -> load_interactions()
  -> build_item_similarity()
  -> score_candidates_for_user()
  -> build_user_item_matrix()
  -> item_similarity_points()
  -> 返回给前端
```

前端拿到数据后，会展示：

- 推荐列表。
- 推荐解释。
- 用户-物品矩阵。
- 共现矩阵。
- 相似度矩阵。
- 推荐链路。
- 匹配过程画布。
- 3D 物品相似度空间。

## 12. 当前系统为什么每次请求都重新计算

生产系统通常不会每次请求都完整重算相似度矩阵。

但当前项目是学习系统，为了让你立刻看到变化：

```text
新增喜欢/删除喜欢
  -> 写入 PostgreSQL
  -> 重新请求 /api/state
  -> 重新计算矩阵和推荐
  -> 前端马上刷新
```

这样最适合学习。

因为你每改一条行为，都能看到：

- 用户-物品矩阵怎么变。
- 共现矩阵怎么变。
- 相似度矩阵怎么变。
- 推荐结果怎么变。
- 推荐解释怎么变。
- 画布连线怎么变。

生产系统会更复杂，通常会分成离线和在线两部分。

## 13. 生产系统里 ItemCF 怎么训练

真实业务里，常见架构是：

```text
用户行为日志
  -> 数据仓库
  -> 离线任务每天/每小时计算物品相似度
  -> 保存 item_sim 表
  -> 在线服务读取用户最近行为
  -> 用相似度表快速召回候选
  -> 排序服务进一步排序
  -> 返回推荐结果
```

### 13.1 离线训练

离线训练一般定时执行。

比如：

```text
每天凌晨 2 点全量训练一次
每小时增量更新一次热门物品相似度
```

训练结果可以存成：

```text
item_id, similar_item_id, similarity
```

例如：

| item_id | similar_item_id | similarity |
| --- | --- | ---: |
| i8 | i14 | 0.667 |
| i8 | i19 | 0.408 |
| i17 | i18 | 0.500 |

这张表可以叫：

```text
item_similarity
```

### 13.2 在线推荐

用户打开页面时：

```text
读取用户最近喜欢/点击/购买的物品
  -> 查这些物品的相似物品
  -> 合并候选
  -> 累加分数
  -> 过滤已看/已买/不合规物品
  -> 排序返回
```

在线推荐不适合慢慢重算全量矩阵。

它需要快。

所以生产系统会提前训练好相似度表。

### 13.3 为什么当前系统不先做离线表

因为现在目标是学习。

当前数据量小，实时重算更直观。

等你理解完整链路后，可以升级成：

```text
新增 item_similarity 表
新增 /api/train 接口
点击训练按钮后写入相似度表
推荐接口直接读取 item_similarity
```

这样就更接近生产系统。

## 14. “训练模型”按钮应该怎么设计

如果后续要在系统里做真正的“训练台”，可以这样设计。

### 14.1 输入

训练输入：

- 用户表。
- 物品表。
- 行为表。
- 时间范围。
- 行为类型。
- 最小共现次数。
- 每个物品保留 Top K 相似物品。

### 14.2 训练过程

训练步骤：

```text
1. 读取 interactions
2. 清洗无效行为
3. 构造 item_user_sets
4. 统计 co_count
5. 计算 sim(i, j)
6. 过滤低相似度关系
7. 每个物品只保留 Top K 相似物品
8. 写入 item_similarity 表
```

### 14.3 输出

训练输出：

- 总用户数。
- 总物品数。
- 总行为数。
- 共现物品对数量。
- 相似度边数量。
- 每个物品的 Top K 相似物品。
- 训练耗时。
- 训练版本号。

### 14.4 可视化

训练完成后可以展示：

- 相似度矩阵。
- 物品相似 Top K。
- 共现网络图。
- 推荐样例。
- 指标变化。

## 15. ItemCF 怎么评估好不好

推荐系统不能只看页面好不好看，也要看推荐是否有效。

### 15.1 离线评估

离线评估常见做法：

```text
按时间切分数据
  -> 前 80% 行为当训练集
  -> 后 20% 行为当测试集
  -> 用训练集计算相似度
  -> 给用户推荐 Top N
  -> 看推荐是否命中测试集真实行为
```

为什么按时间切分？

因为推荐系统是在预测未来。

如果随机切分，可能会泄漏未来信息。

### 15.2 Precision@K

Precision@K 表示：

```text
推荐的 K 个物品里，有多少比例真的被用户喜欢。
```

公式：

```text
Precision@K = 推荐命中数量 / K
```

例如推荐 5 个，命中 2 个：

```text
Precision@5 = 2 / 5 = 0.4
```

它关心“推荐列表准不准”。

### 15.3 Recall@K

Recall@K 表示：

```text
用户真实喜欢的物品里，有多少被推荐出来。
```

公式：

```text
Recall@K = 推荐命中数量 / 用户测试集真实喜欢数量
```

例如用户测试集中有 4 个真实喜欢物品，推荐命中 2 个：

```text
Recall@5 = 2 / 4 = 0.5
```

它关心“有没有尽量找全”。

### 15.4 HitRate@K

HitRate@K 表示：

```text
只要推荐列表命中至少一个真实喜欢物品，就算这个用户命中。
```

适合看整体用户覆盖。

### 15.5 NDCG@K

NDCG@K 关注排序位置。

同样命中一个物品：

```text
排第 1 名比排第 10 名更好。
```

所以 NDCG 会奖励更靠前的命中。

### 15.6 Coverage

Coverage 表示推荐覆盖了多少物品。

如果系统永远只推荐最热门的 5 个物品，Precision 可能不低，但 Coverage 很差。

### 15.7 Novelty

Novelty 表示推荐是否有新鲜感。

如果总是推荐用户早就知道的热门物品，新颖性会差。

### 15.8 Diversity

Diversity 表示推荐列表内部是否多样。

如果推荐列表全是同一类内容，用户可能会觉得窄。

### 15.9 在线 A/B 实验

最终业务效果还是要线上验证。

常见指标：

- 点击率 CTR。
- 转化率 CVR。
- 收藏率。
- 完播率。
- 购买率。
- 用户停留时长。
- 次日留存。

## 16. 冷启动问题怎么解决

### 16.1 新用户冷启动

新用户没有行为。

ItemCF 不知道他喜欢什么。

解决方式：

- 让用户选择兴趣。
- 推荐热门内容。
- 推荐编辑精选内容。
- 根据注册信息或场景做粗推荐。
- 等用户产生行为后再进入 ItemCF。

### 16.2 新物品冷启动

新物品没有用户行为。

ItemCF 不知道它和谁相似。

解决方式：

- 给新物品一些曝光流量。
- 用内容标签辅助推荐。
- 用 embedding 做内容相似推荐。
- 运营人工配置相似物品。
- 等真实行为积累后再进入 ItemCF。

当前系统里的“自定义物品”就是新物品冷启动案例。

你创建一个新物品并喜欢它后，如果只有你一个人喜欢，它不会马上有强相似关系。

这不是 bug，而是 ItemCF 的真实特性。

## 17. 常见改进方法

### 17.1 时间衰减

用户最近的行为通常更重要。

可以给近期行为更高权重：

```text
preference(u, i) = 行为权重 * 时间衰减系数
```

例如：

```text
今天喜欢 = 1.0
30 天前喜欢 = 0.5
180 天前喜欢 = 0.1
```

### 17.2 行为权重

不同类型行为强度不同。

例如：

```text
浏览 = 1
点赞 = 2
收藏 = 3
购买 = 5
```

然后推荐分变成：

```text
score(u, j) = sum(sim(j, i) * weight(u, i))
```

### 17.3 最小共现过滤

如果两个物品只被 1 个用户共同喜欢，相似度可能不稳定。

可以设置：

```text
co_count(i, j) >= 2
```

才保留相似关系。

数据越大，这个阈值越有意义。

### 17.4 Top K 相似物品截断

生产系统里，不会保存每个物品和所有物品的相似度。

通常只保留每个物品最相似的 K 个：

```text
每个物品保留 Top 50 相似物品
```

这样能减少存储和在线计算量。

### 17.5 惩罚热门物品

即使用了余弦归一化，热门物品仍可能过强。

可以加入流行度惩罚：

```text
score = 原始分 / log(物品热度 + 1)
```

### 17.6 和内容推荐混合

ItemCF 看行为，不看内容。

内容推荐看标题、标签、文本、图片 embedding。

实际系统常常混合：

```text
ItemCF 召回
内容相似召回
热门召回
新物品召回
人工运营召回
```

然后把多路候选合并排序。

## 18. ItemCF 和 RAG/向量数据库有什么区别

这个点很重要。

### 18.1 RAG 的相似度

RAG 通常是：

```text
文本切片
  -> embedding 模型
  -> 向量数据库
  -> 用问题向量检索相似文本
```

相似度来自文本语义。

例如：

```text
“RAG 怎么切片”
```

会匹配到讲文档切片的段落。

### 18.2 ItemCF 的相似度

ItemCF 是：

```text
用户行为
  -> 共现统计
  -> 物品相似度
```

相似度来自行为。

例如：

```text
很多用户同时喜欢 RAG 系统实战 和 Chroma 向量数据库
```

系统就认为它们相似。

### 18.3 为什么 ItemCF 不用向量数据库

当前 ItemCF 不需要向量数据库，因为它没有做文本 embedding 检索。

它要存的是：

- 用户。
- 物品。
- 行为。
- 共现。
- 相似度。

这些用 PostgreSQL 就够。

如果以后做混合推荐，可以把内容 embedding 加进来。

那时候可能会同时使用：

```text
PostgreSQL 存行为
向量数据库存内容 embedding
推荐服务融合两路结果
```

## 19. 你应该怎么用当前系统学习

建议按这个顺序操作。

### 19.1 第一步：看默认用户

打开 ItemCF 学习实验室，进入：

```text
推荐可视化台
```

先看当前用户喜欢了哪些物品。

再看推荐结果。

重点观察：

- 推荐物品是什么。
- 推荐分是多少。
- 哪些历史物品贡献了分数。
- 画布上的橙色路径从哪里到哪里。

### 19.2 第二步：删除一个喜欢

删除当前用户的一个已喜欢物品。

观察：

- 推荐结果是否变化。
- 当前推荐解释是否变化。
- 矩阵是否变化。
- 画布路径是否变化。

你要建立一个感觉：

```text
用户行为变了，推荐结果就会变。
```

### 19.3 第三步：新增一个喜欢

选择一个新物品，加到当前用户喜欢。

观察：

- 它会不会影响候选得分。
- 哪些候选物品被推高。
- 推荐分贡献是否多了一项。

### 19.4 第四步：创建自定义物品

创建一个新物品。

你会发现它刚开始不一定产生强推荐效果。

这正好用来理解：

```text
新物品冷启动。
```

### 19.5 第五步：看矩阵可视化

进入：

```text
矩阵可视化
```

重点看三张矩阵：

- 用户-物品矩阵。
- 共现矩阵。
- 相似度矩阵。

你要能把页面上的推荐结果反推回矩阵。

这才算真正看懂 ItemCF。

## 21. 一句话总结

```text
ItemCF 的训练，就是把用户行为日志变成物品相似度矩阵。

ItemCF 的推荐，就是用用户已喜欢物品去相似度矩阵里找候选物品，
再累加相似度得分，过滤已喜欢物品，输出 Top N。
```

你现在这个系统，就是把这条链路拆开给你看。
