## 2. 数据为什么不用向量数据库

ItemCF 的相似度不是文本语义相似度，而是用户行为共现。

例如：

```text
很多用户同时喜欢 A 和 B
  -> A 和 B 相似
```

所以核心数据是关系型数据：

- 用户。
- 物品。
- 行为。

这些适合放关系型数据表或浏览器本地结构化数据。

Chroma/向量数据库更适合 RAG 的文本 embedding 检索，不适合存 ItemCF 的原始行为数据。

## 4. 历史后端接口

| 接口 | 作用 |
| --- | --- |
| `GET /health` | 检查 PG 是否可用，并返回用户、物品、行为数量。 |
| `GET /api/state?user_id=u2` | 从 PG 读取数据，计算矩阵、推荐、3D 点位。 |
| `POST /api/items` | 创建自定义物品，可同时加入当前用户喜欢。 |

| `DELETE /api/interactions` | 删除当前用户的一条喜欢行为，写入 PG。 |
| `POST /api/reset?user_id=u2` | 重置学习数据。 |

## 5. 核心算法

### 5.1 用户-物品矩阵

根据 interactions 构造：

```text
用户喜欢过物品 -> 1
没有行为 -> 0
```

### 5.2 物品相似度

公式：

```text
sim(i, j) = co_count(i, j) / sqrt(count(i) * count(j))
```

其中：

- `co_count(i, j)`：同时喜欢 i 和 j 的用户数。
- `count(i)`：喜欢 i 的用户数。
- `count(j)`：喜欢 j 的用户数。

### 5.3 推荐分

公式：

```text
score(u, j) = sum(sim(j, i) * preference(u, i))
```

当前 `preference(u, i)` 是 1，因为只有喜欢行为。

## 6. 推荐可视化台

推荐可视化台用于把 ItemCF 的匹配过程讲清楚。

它没有新增算法，也没有新增假数据，而是复用当前实验数据和推荐结果。

页面分成三块：

- 目标用户：展示当前用户和已喜欢物品。
- 匹配过程画布：用 Canvas 画出用户、历史物品、候选物品之间的连线。
- 推荐结果：展示 Top N，并在点击推荐项后高亮主要贡献路径。

目标用户区域支持直接编辑行为：

- 新增喜欢：选择一个当前用户还没喜欢过的物品。
- 自定义物品：创建一个新物品，并自动加入当前用户喜欢。
- 删除喜欢：从当前用户已喜欢列表里删除一条行为。
- 每次新增或删除后，推荐、矩阵、画布都会重新计算。

自定义物品是推荐系统里的冷启动案例。ItemCF 不是根据标题内容推荐，而是根据用户行为共现推荐。因此新物品刚创建时，如果只有当前用户喜欢它，它和其他物品的共现关系很少，通常不会立刻成为强推荐。让多个用户都喜欢过它后，它才会产生稳定相似度。

连线含义：

- 用户到历史物品：这个用户真实喜欢过这些物品。
- 历史物品到候选物品：两个物品因为用户共现产生相似度。
- 橙色连线：当前选中推荐的主要贡献来源。
- 绿色淡线：其他有共现关系的候选路径。

鼠标悬停到画布节点时，会显示完整标题和节点说明，避免长标题被圆点里的短文本截断。

下方的推荐链路会把过程拆成：

```text
读取行为 -> 寻找相似 -> 累加得分 -> 过滤排序
```

## 7. 3D 物品相似度空间

前端用 Canvas 绘制 3D 教学视图。

每个点代表一个物品。

连线代表两个物品相似度大于 0。

连线越亮，说明两个物品越经常被同一批用户共同喜欢。

点击推荐项后，3D 视图会高亮该推荐物品和贡献来源物品之间的关系。

注意：

```text
这是 ItemCF 相似度空间，不是 Chroma 向量空间。
```
