# 技术架构规划

## 1. 当前架构

第一版使用纯静态前端：

```text
index.html
  -> styles.css
  -> app.js
      -> 内置 users/items/interactions
      -> 计算共现矩阵
      -> 计算相似度矩阵
      -> 计算用户推荐分
      -> 渲染页面
```

这样做的原因：

- ItemCF 的核心学习难点在计算逻辑，不在服务端框架。
- 小数据集可以在浏览器里直接计算，便于观察。
- 先把算法讲透，再接后端更稳。

## 2. 算法模块

`app.js` 里保留几个清晰函数：

- `buildUserItemMatrix()`：构造用户-物品矩阵。
- `buildItemSimilarity()`：计算物品相似度。
- `recommendForUser()`：为目标用户计算推荐榜单。
- `render*()`：负责页面展示。

第一版不抽成复杂模块，避免学习者在工程结构里迷路。

## 3. 后续后端形态

后续可以扩展为：

```text
前端
  -> FastAPI
      -> /users
      -> /items
      -> /interactions
      -> /recommendations/{user_id}
  -> SQLite / PostgreSQL
```

推荐计算可以先在请求时实时计算，再逐步演进到离线批处理：

```text
交互日志
  -> 离线计算相似度矩阵
  -> 保存 item_similarity
  -> 在线读取用户历史
  -> 合并推荐分
```

## 4. 工程边界

当前不引入框架、不打包、不依赖 npm。

只有当需要：

- 数据上传。
- 多页面路由。
- 后端 API。
- 自动测试。
- 真实数据规模。

才进入下一阶段。
