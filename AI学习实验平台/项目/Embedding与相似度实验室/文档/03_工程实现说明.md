# 工程实现说明

## 1. 项目结构

```text
Embedding与相似度实验室/
  README.md
  需求/
  规划/
  文档/
  学习资料/
  前端原型/
  后端/
```

## 2. 后端接口

后端默认端口：

```text
8030
```

接口：

| 接口 | 作用 |
| --- | --- |
| `GET /health` | 检查后端是否可用。 |
| `GET /api/examples` | 返回默认查询和候选文本。 |
| `POST /api/compare` | 计算 embedding、相似度和 Top K。 |

## 3. POST /api/compare 输入

```json
{
  "query": "RAG 里的 embedding 和向量数据库是什么关系？",
  "metric": "cosine",
  "topK": 5,
  "candidates": [
    {
      "id": "d1",
      "text": "RAG 使用 embedding 把问题和文档片段变成向量，再计算相似度完成检索。"
    }
  ]
}
```

## 4. 输出

返回：

- query embedding。
- candidate embedding。
- token 列表。
- token 到维度的映射。
- 向量数组。
- 相似度分数。
- Top K 排序。
- 解释文案。

## 5. 为什么前端有离线兜底

学习时后端可能没启动。

为了不让页面空白，前端内置了同样思路的简化算法。

如果后端不可用，状态会显示：

```text
离线教学算法
```

这时仍然可以学习流程，但正式验证以后端为准。

## 6. 当前没有数据库的原因

本项目学习的是：

```text
embedding 计算和相似度比较
```

暂时不学习：

```text
向量持久化和索引查询
```

所以当前不引入 PostgreSQL、Chroma、FAISS。

向量库会单独做一个项目。

```text
查询向量
```

## 8. 查询文本、查询向量和候选集合

页面里的查询文本，就是你要拿去检索的问题。

它查询的不是互联网，也不是整个项目文件夹，而是当前页面里的：

```text
候选文本集合
```

候选文本集合可以理解成一个迷你知识库。

真实 RAG 里，这个集合会变成向量数据库里的文档 chunk。

查询文本向量，就是系统把查询文本转换到一组语义维度后的数字数组。

当前教学维度包括：

- RAG 检索。
- Embedding 向量化。
- 向量数据库。
- 相似度计算。
- 文档切分。
- Rerank 重排。
- Agent 工具。
- 推荐算法。

如果候选文本和查询文本在这些维度上都很强，它们的相似度就会更高。

## 9. 当前教学算法边界

当前语义维度 embedding 只是教学模型。

它能帮助你看懂：

- token。
- 向量维度。
- 归一化。
- 相似度。
- Top K。

但它不能替代真实 embedding 模型。

真实项目应该使用：

- OpenAI embeddings。
- Qwen embeddings。
- BGE。
- bge-m3。
- text2vec。
- 其他生产级 embedding 模型。
