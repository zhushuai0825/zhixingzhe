# 技术架构规划

## 1. 总体架构

```text
前端原型
  -> 调用 FastAPI /api/compare
  -> 后端本地计算 embedding 和相似度
  -> 返回 tokens、vectors、scores
  -> 前端渲染向量条形图和匹配画布
```

## 2. 前端

技术：

- HTML
- CSS
- JavaScript
- Canvas

前端职责：

- 输入查询文本。
- 编辑候选文本。
- 选择相似度算法。
- 展示 token。
- 展示向量条形图。
- 展示 Top K 结果。
- 绘制匹配过程画布。
- 后端不可用时使用本地算法兜底。

## 3. 后端

技术：

- Python
- FastAPI
- Pydantic

后端职责：

- 提供教学用 embedding 函数。
- 提供相似度计算函数。
- 返回完整计算过程。
- 保持结果可解释。

## 4. 数据存储

当前不需要 PostgreSQL，也不需要向量数据库。

原因：

```text
本项目学习 embedding 和相似度计算本身，
不是学习向量持久化。
```

后续“向量库实验室”会单独学习：

- Chroma。
- FAISS。
- pgvector。
- Milvus。

## 5. 教学 embedding 方法

当前使用语义维度 embedding：

```text
文本清洗
  -> 分词/tokenize
  -> token 映射到有名字的教学语义维度
  -> 统计语义维度权重
  -> L2 归一化
```

优点：

- 本地运行。
- 可解释。
- 不需要 API Key。
- 方便显示每个 token 对语义维度的影响。

缺点：

- 不是深度语义模型。
- 无法真正理解复杂语义。
- 只是入门教学模型。

## 6. 后续项目拆分

推荐拆成这些独立项目：

1. Embedding 与相似度实验室。
2. 文本切分实验室。
3. 向量库实验室。
4. Rerank 重排序实验室。
5. RAG 检索链路整合实验室。

每个项目都独立有前端、后端、文档，再接入统一平台。
