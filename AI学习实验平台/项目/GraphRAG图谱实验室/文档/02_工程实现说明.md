# Graph RAG 工程实现说明

## 1. 当前版本定位

当前是纯前端教学版，重点是把 Graph RAG 的数据流展示清楚。

它不是生产级图数据库，也没有调用真实大模型。

## 2. 数据结构

### 2.1 实体

```js
{ id, name, type, note }
```

含义：

- `id`：前端唯一标识。
- `name`：实体名。
- `type`：实体类型。
- `note`：学习解释。

### 2.2 关系

```js
{ from, label, to, weight }
```

含义：

- `from`：起点实体。
- `label`：关系名称。
- `to`：终点实体。
- `weight`：教学版关系强度。

### 2.3 证据片段

```js
{ id, title, source, entities, text }
```

含义：

- `title`：片段标题。
- `source`：来源文件。
- `entities`：这个片段关联哪些实体。
- `text`：原文证据。

## 3. 查询流程

1. 读取用户问题。
2. 根据实体名、类型、说明做起点命中。
3. 从起点实体出发，按最大跳数遍历关系。
4. 给路径计算教学分数。
5. 根据路径里的实体找回相关 chunk。
6. 生成回答草稿和证据列表。
7. 把命中实体、路径、证据同步到 3D 图谱。

## 4. 可视化实现

3D 画布使用本地 `three.module.js`。

视觉含义：

- 蓝色节点：普通实体。
- 橙色节点：问题命中的起点实体。
- 绿色节点：路径命中的实体。
- 深色节点：当前选中实体。
- 橙色线：当前检索路径命中的关系。

交互：

- 鼠标拖拽旋转。
- 滚轮缩放。
- 悬停节点查看实体解释。
- 点击节点查看关系和证据详情。

## 5. 后续升级方向

生产级可以升级为：

- 后端使用 FastAPI。
- 节点和关系存 PostgreSQL 或 Neo4j。
- 实体抽取用大模型结构化输出。
- 关系抽取用 Prompt 或专门模型。
- 结合向量检索和 BM25 做多路召回。
- 用 Rerank 合并路径证据和文本证据。
- 用 RAG 评测实验室的方法评估路径命中率和回答忠实度。

## 6. 真实后端接口建议

如果后续要把这个实验室升级成真实项目，可以先设计这些接口：

```text
POST /api/graph/entities
GET  /api/graph/entities
POST /api/graph/relations
GET  /api/graph/relations
POST /api/graph/chunks
POST /api/graph/search
```

其中 `/api/graph/search` 的请求可以是：

```json
{
  "query": "Graph RAG 为什么适合金融风控？",
  "max_hops": 2,
  "relation_types": ["支持", "适合", "依赖"],
  "top_k_paths": 5
}
```

返回可以是：

```json
{
  "seeds": ["Graph RAG", "金融风控"],
  "paths": [
    {
      "nodes": ["Graph RAG", "知识图谱", "多跳推理", "金融风控"],
      "relations": ["构建于", "支持", "适合"],
      "score": 0.86
    }
  ],
  "evidence_chunks": [
    {
      "source": "risk_control.md",
      "text": "金融风控需要把客户、设备、交易和风险事件关联起来分析。"
    }
  ]
}
```

## 7. PostgreSQL 表设计草案

如果暂时不用 Neo4j，可以用 PostgreSQL 先模拟：

```sql
create table graph_entities (
  id text primary key,
  name text not null,
  type text not null,
  description text,
  aliases text[] default '{}'
);

create table graph_relations (
  id text primary key,
  from_entity_id text references graph_entities(id),
  relation_type text not null,
  to_entity_id text references graph_entities(id),
  confidence numeric not null default 0.7,
  source_chunk_id text
);

create table graph_chunks (
  id text primary key,
  source text not null,
  text text not null,
  entity_ids text[] default '{}',
  metadata jsonb default '{}'
);
```

这种设计适合学习和小规模实验。后续如果路径查询复杂，再迁到图数据库。

## 8. 真实系统的关键坑

- 实体重复：同一个实体被抽成多个名字，需要实体对齐。
- 实体歧义：同名不同物，需要消歧。
- 关系错误：大模型抽取关系可能误判，需要保存来源证据。
- 路径爆炸：跳数一多，候选路径会变很多，需要限制扩散。
- 证据缺失：只有图谱路径，没有原文 chunk，不能直接生成答案。
- 评测困难：要准备多跳问题和标准路径，不能只看最终答案。
