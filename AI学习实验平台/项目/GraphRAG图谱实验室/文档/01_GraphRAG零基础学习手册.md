# Graph RAG 零基础学习手册

## 1. Graph RAG 是什么

普通 RAG 的核心是：

```text
问题 -> 向量化 -> 找相似 chunk -> 拼上下文 -> 大模型回答
```

Graph RAG 的核心是：

```text
文档 -> 抽实体 -> 抽关系 -> 建图谱
问题 -> 命中实体 -> 沿关系找路径 -> 找回原文证据 -> 大模型回答
```

你可以把普通 RAG 理解成“在一堆文本片段里找相似内容”，把 Graph RAG 理解成“先把知识整理成关系网，再沿关系网找证据”。

## 2. 为什么需要 Graph RAG

普通向量 RAG 很强，但它有一个明显短板：它主要看“这段文字和问题像不像”。

如果答案需要多个片段共同支撑，普通 RAG 可能会漏掉关系。

例子：

```text
问题：Graph RAG 为什么适合金融风控？
```

可能有两个片段：

```text
片段 A：Graph RAG 可以沿实体关系做多跳推理。
片段 B：金融风控需要把客户、设备、交易、账户、风险事件关联分析。
```

单看任何一个片段，都不一定完整回答问题。Graph RAG 会把它们变成关系：

```text
Graph RAG -> 支持 -> 多跳推理
多跳推理 -> 适合 -> 金融风控
```

这样系统就能通过路径说明：Graph RAG 适合金融风控，是因为金融风控需要跨实体关联，而 Graph RAG 支持多跳推理。

## 3. 实体是什么

实体就是知识里的重要对象。

例子：

- Graph RAG。
- 知识图谱。
- 金融风控。
- 法律条文。
- Neo4j。
- 某个客户。
- 某笔交易。
- 某台设备。

实体不一定是人名、地名，也可以是技术名、业务场景、系统模块、产品功能。

## 4. 关系是什么

关系就是实体之间的连接。

例子：

```text
Graph RAG --构建于--> 知识图谱
Graph RAG --需要--> 实体抽取
知识图谱 --支持--> 多跳推理
多跳推理 --适合--> 金融风控
```

关系有三个核心字段：

```text
起点实体、关系名称、终点实体
```

这也叫三元组：

```text
(Graph RAG, 构建于, 知识图谱)
```

## 5. 多跳推理是什么

一跳：

```text
Graph RAG -> 知识图谱
```

两跳：

```text
Graph RAG -> 知识图谱 -> 多跳推理
```

三跳：

```text
Graph RAG -> 知识图谱 -> 多跳推理 -> 金融风控
```

跳数越多，能找到的关联越远，但噪声也越多。生产系统通常会限制最大跳数、关系类型和路径数量。

## 6. Graph RAG 的完整流程

### 6.1 文档解析

先把 PDF、Word、网页或 Markdown 解析成文本。

### 6.2 文本切片

长文档仍然要切成 chunk，因为最终回答要引用原文证据。

### 6.3 实体抽取

从 chunk 里找重要对象。

可以用：

- 规则。
- NER 模型。
- 大模型结构化抽取。

### 6.4 关系抽取

从句子里判断实体之间有什么关系。

例如：

```text
Graph RAG 可以把文档构建成知识图谱。
```

可以抽成：

```text
Graph RAG --构建成--> 知识图谱
```

### 6.5 图谱入库

真实系统里可以存：

- Neo4j。
- NebulaGraph。
- TuGraph。
- PostgreSQL 节点表和关系表。

学习阶段可以先用前端对象模拟。

### 6.6 查询时命中实体

用户提问后，系统先判断问题里有哪些实体。

比如：

```text
Graph RAG 为什么适合法律？
```

命中实体：

```text
Graph RAG、法律条文
```

### 6.7 图遍历

从命中的实体出发，找 1 到 3 跳路径。

### 6.8 证据回填

路径只能说明“实体有关联”，但不能直接当答案。

最终还要找回包含这些实体的原文 chunk，让大模型基于原文回答。

## 7. Graph RAG 和向量 RAG 怎么配合

企业里一般不是二选一，而是组合：

```text
向量检索：找语义相似片段
BM25：找关键词强匹配片段
图谱检索：找实体关系路径
Rerank：把多路结果重新排序
LLM：基于证据生成答案
```

Graph RAG 是增强检索的一种方式，不是替代所有 RAG。

## 8. 什么时候不该用 Graph RAG

如果你的场景只是简单 FAQ：

```text
请问报销流程是什么？
```

普通 RAG 就够了。

如果强行上 Graph RAG，会多出实体抽取、关系抽取、图谱维护、路径评测等成本。

Graph RAG 更适合：

- 问题需要多跳关联。
- 实体关系很重要。
- 答案来自多个文档。
- 需要解释推理链路。

## 9. 常见面试追问

### Graph RAG 的优势是什么？

它能把分散在多个文档里的实体关系串起来，适合复杂关联问题。比如风控里要把客户、设备、交易、账户串起来，普通向量检索可能只找到某一个片段，Graph RAG 可以找到关系路径。

### Graph RAG 的缺点是什么？

构建成本高，实体和关系抽取可能出错，图谱更新麻烦，多跳路径容易引入噪声，评测也更复杂。

### Graph RAG 需要训练大模型吗？

不一定。多数应用是用现有大模型或信息抽取模型做实体关系抽取，再用图数据库检索。你更需要学数据建模、抽取、图检索、证据回填和评测。

### Graph RAG 怎么防幻觉？

不要让模型只看图谱边回答。必须把路径关联到原文 chunk，让模型基于引用回答。证据不足时拒答。

## 10. 用一个完整例子手算 Graph RAG

假设你有三段资料：

```text
资料 1：Graph RAG 会把文档中的实体和关系构造成知识图谱。
资料 2：知识图谱可以支持多跳推理。
资料 3：金融风控需要把客户、设备、交易和风险事件关联起来分析。
```

第一步，抽实体：

```text
Graph RAG
文档
实体
关系
知识图谱
多跳推理
金融风控
客户
设备
交易
风险事件
```

第二步，抽关系：

```text
Graph RAG --构造--> 知识图谱
知识图谱 --支持--> 多跳推理
金融风控 --需要--> 关联分析
关联分析 --关联--> 客户
关联分析 --关联--> 设备
关联分析 --关联--> 交易
关联分析 --关联--> 风险事件
```

第三步，用户提问：

```text
Graph RAG 为什么适合金融风控？
```

第四步，命中起点实体：

```text
Graph RAG
金融风控
```

第五步，找路径。

如果图谱里只有这些关系，系统可能找到：

```text
Graph RAG -> 知识图谱 -> 多跳推理
金融风控 -> 关联分析 -> 客户/设备/交易/风险事件
```

这时候还差一条桥：

```text
多跳推理 --适合--> 关联分析
```

加上这条桥以后，就能形成路径：

```text
Graph RAG -> 知识图谱 -> 多跳推理 -> 关联分析 -> 金融风控
```

这就是 Graph RAG 的一个关键点：它不是魔法。图谱里关系质量越好，路径越能支撑答案；关系缺失，系统也会找不到好路径。

## 11. Graph RAG 的数据通常怎么存

学习阶段你可以先用三个表理解。

### 11.1 实体表

```text
graph_entities
```

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| id | 实体 ID | e1 |
| name | 实体名称 | Graph RAG |
| type | 实体类型 | 技术架构 |
| aliases | 别名 | 图谱 RAG |
| description | 解释 | 基于实体关系增强检索 |

### 11.2 关系表

```text
graph_relations
```

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| id | 关系 ID | r1 |
| from_entity_id | 起点实体 | e1 |
| relation_type | 关系类型 | 支持 |
| to_entity_id | 终点实体 | e5 |
| confidence | 关系置信度 | 0.86 |
| source_chunk_id | 来自哪个 chunk | c12 |

### 11.3 证据表

```text
graph_chunks
```

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| id | chunk ID | c12 |
| source | 来源文件 | risk_control.md |
| text | 原文片段 | 金融风控需要关联分析 |
| entity_ids | 关联实体 | e1,e5,e9 |
| embedding | 向量 | [0.12, 0.08, ...] |

你要记住：Graph RAG 不是只存图。它还要保留原文 chunk，因为最后回答要可引用。

## 12. Neo4j、PostgreSQL、向量库分别放什么

一个真实系统可以这样分工：

| 组件 | 主要存什么 | 作用 |
| --- | --- | --- |
| PostgreSQL | 文档、用户、任务、chunk metadata | 业务主库，保证数据可靠 |
| 向量库/pgvector | chunk embedding | 做语义相似检索 |
| Neo4j/NebulaGraph | 实体节点、关系边 | 做路径查询和多跳推理 |
| Redis | 缓存热门查询、会话状态 | 降低延迟 |

如果你是初学者，不要一开始就装一堆东西。你可以按顺序来：

```text
先用前端对象理解图谱
再用 PostgreSQL 表模拟节点和关系
再学 Neo4j 这种专业图数据库
最后做向量检索 + 图谱检索 + Rerank 的融合
```

## 13. 实体和关系怎么抽取

真实项目里常见三种方式。

### 13.1 规则抽取

适合格式非常固定的文档。

例子：

```text
产品：A 系统
依赖：PostgreSQL、Redis
```

规则可以直接抽：

```text
A 系统 --依赖--> PostgreSQL
A 系统 --依赖--> Redis
```

优点是稳定、便宜。

缺点是泛化差，换个写法可能就抽不到。

### 13.2 模型抽取

可以用 NER 模型抽实体，用关系抽取模型抽关系。

优点是比规则灵活。

缺点是你要找模型、部署模型、调效果。

### 13.3 大模型结构化抽取

用大模型把文本转成 JSON。

示例 Prompt 思路：

```text
请从下面文本中抽取实体和关系。
实体字段：name, type, description
关系字段：from, relation, to, evidence
只输出 JSON，不要输出解释。
```

大模型输出示例：

```json
{
  "entities": [
    { "name": "Graph RAG", "type": "技术架构" },
    { "name": "知识图谱", "type": "数据结构" }
  ],
  "relations": [
    {
      "from": "Graph RAG",
      "relation": "构建于",
      "to": "知识图谱",
      "evidence": "Graph RAG 会把文档中的实体和关系构造成知识图谱"
    }
  ]
}
```

注意：大模型抽取也会错，所以要保存 `evidence`，方便回查原文。

## 14. Graph RAG 检索时到底发生了什么

用户问：

```text
Graph RAG 为什么适合法律条文？
```

系统可以分成 6 步：

1. 识别问题中的实体：`Graph RAG`、`法律条文`。
2. 查实体表，找到它们的节点 ID。
3. 从这些节点出发，找 1 到 3 跳关系。
4. 过滤无关关系，例如只保留 `适合`、`支持`、`依赖`、`属于`。
5. 找到路径关联的 chunk。
6. 把路径、chunk、引用来源一起给大模型。

最终给模型的上下文可以长这样：

```text
问题：Graph RAG 为什么适合法律条文？

图谱路径：
Graph RAG -> 支持 -> 多跳推理
多跳推理 -> 适合 -> 法律条文

证据：
[legal_graph.md] 法律问答经常要连接法条、案例、主体、行为、责任等多个实体。Graph RAG 可以沿关系路径找到多个证据点。

要求：
只能基于证据回答；如果证据不足，必须说明不足。
```

## 15. Graph RAG 和 Rerank 的关系

Graph RAG 找到路径以后，不代表所有路径都好。

常见问题：

- 路径太长。
- 关系太弱。
- 路径里有无关实体。
- 证据 chunk 只是提到实体，但没有回答问题。

所以真实系统通常还会 Rerank。

Rerank 可以把候选分成：

```text
候选 1：向量检索 chunk
候选 2：BM25 chunk
候选 3：图谱路径关联 chunk
候选 4：图谱路径摘要
```

然后统一排序，选最能回答问题的证据进入 prompt。

## 16. Graph RAG 怎么评测

只看“答案对不对”不够。Graph RAG 至少要看这些指标。

### 16.1 实体命中率

问题里的关键实体有没有识别出来。

例如问题里有 `Graph RAG` 和 `金融风控`，系统只识别出 `Graph RAG`，就不够。

### 16.2 关系准确率

抽出来的关系是否正确。

例如：

```text
Graph RAG --替代--> 向量 RAG
```

这就可能是错误关系，因为 Graph RAG 更多是补充向量 RAG，而不是完全替代。

### 16.3 路径命中率

标准答案需要的路径有没有被找回来。

### 16.4 路径噪声率

找到的路径里有多少是无关路径。

### 16.5 证据忠实度

最终答案是否真的被原文 chunk 支持。

Graph RAG 很容易让人误以为“路径看起来合理，所以答案就可靠”。不对，最终仍然要看原文证据。

## 17. 你应该怎么练

第一轮练习：

1. 打开 Graph RAG 图谱实验室。
2. 用默认问题运行一次。
3. 看命中了哪些实体。
4. 看路径是几跳。
5. 看证据片段是不是支撑回答。

第二轮练习：

1. 新增实体：`医疗知识库`。
2. 新增关系：`Graph RAG --适合--> 医疗知识库`。
3. 新增证据：写一段医疗知识库需要关联症状、疾病、药品、禁忌的文字。
4. 提问：`Graph RAG 为什么适合医疗知识库？`
5. 看路径和证据是否变化。

第三轮练习：

1. 把最大跳数改成 1。
2. 再改成 2。
3. 再改成 3。
4. 对比路径数量和噪声。

你要形成一个直觉：跳数越多，召回越广，但越容易乱。

## 18. 大厂面试题补充

### 18.1 如果让你设计一个企业 Graph RAG，你会怎么设计？

可以这样答：

先做文档解析和切分，保存 chunk 和 metadata。然后做实体抽取和关系抽取，把实体存节点表，关系存边表，并保留每条关系来自哪个 chunk。查询时先做实体识别，再做向量检索、关键词检索和图谱路径检索，多路召回后用 Rerank 排序，最后把高质量证据交给大模型。回答时必须带引用，证据不足要拒答。评测上看实体命中率、路径命中率、Recall@K、忠实度和拒答正确率。

### 18.2 Graph RAG 为什么不一定比普通 RAG 好？

因为 Graph RAG 的效果依赖实体和关系抽取质量。如果抽错关系，系统会沿错误路径找证据。它还会增加构图成本、更新成本和查询复杂度。简单 FAQ 用普通 RAG 更划算。

### 18.3 Graph RAG 如何处理图谱更新？

新增文档后要重新解析、切片、抽实体、抽关系，然后做实体对齐。已有实体不要重复创建，要合并别名和同义词。删除文档时，要删除或降低对应关系的置信度。生产系统里关系最好带 `source_chunk_id`，这样能追溯来源。

### 18.4 图谱里实体重名怎么办？

要做实体消歧。比如 `苹果` 可能是水果，也可能是公司。可以用实体类型、上下文、来源领域、别名和相邻关系判断。必要时保留两个不同实体：`苹果公司` 和 `苹果水果`。

### 18.5 Graph RAG 和 Agent 有什么关系？

Graph RAG 是一种检索能力，Agent 是任务执行框架。Agent 可以把图谱检索当作一个工具：当它发现问题需要多跳关系时调用 Graph RAG；如果只是简单事实问题，就调用普通向量检索。

### 18.6 生产中如何避免多跳路径爆炸？

限制最大跳数，限制关系类型，设置路径分数阈值，限制每层扩散数量，使用业务规则过滤无关实体，并对路径结果做 Rerank。
