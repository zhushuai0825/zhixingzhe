# RAG 实验室需求

## 1. 模块目标

RAG 实验室是知行者 AI 实验室的第一核心模块。

目标是让用户从 0 基础理解并跑通 RAG 的完整工程链路：

```text
文档上传
  -> 文档解析
  -> 文本清洗
  -> 文本切片
  -> Embedding
  -> 向量数据库
  -> 检索
  -> Rerank
  -> 拼接上下文
  -> 大模型生成
  -> 引用展示
  -> 自动评测
```

## 2. 学习目标

用户通过 RAG 实验室要学会：

- RAG 是什么
- 为什么 RAG 能减少幻觉
- 文档为什么要切片
- chunk size 和 overlap 的影响
- Embedding 在 RAG 里的作用
- 向量数据库存什么
- 向量检索如何匹配问题和片段
- BM25 和向量检索有什么区别
- Hybrid 检索为什么常用
- Rerank 为什么能提高准确率
- 无依据拒答为什么重要
- RAG 怎么做自动评测

## 3. 页面结构

RAG 实验室页面建议采用三栏结构：

```text
左侧：RAG 流程步骤
中间：当前步骤实验区
右侧：解释面板 / 观察点 / 常见错误
```

### 3.1 左侧流程步骤

步骤包括：

1. 文档解析
2. 文本清洗
3. 文本切片
4. Embedding
5. 向量入库
6. 检索
7. Rerank
8. Prompt 拼接
9. 生成回答
10. 引用与拒答
11. 自动评测

### 3.2 中间实验区

根据当前步骤展示不同实验：

- 文档解析：展示原始文件和提取文本
- 文本切片：展示 chunk 列表
- Embedding：展示向量维度和局部数值
- 向量入库：展示 chunk 和 vector 的关系
- 检索：展示问题、Top K 命中、相似度
- Rerank：展示重排前后顺序
- 生成回答：展示 Prompt、引用和回答
- 自动评测：展示用例、预期、实际、通过率

### 3.3 右侧解释面板

每一步都要解释：

- 这一步解决什么问题
- 输入是什么
- 输出是什么
- 常见参数
- 常见错误
- 实际企业中怎么用

## 4. RAG 类型

系统要讲清楚四种主线：

### 4.1 Naive RAG

流程：

```text
分块 -> 向量化 -> 向量检索 -> 拼上下文 -> 生成
```

适合：

- 入门学习
- FAQ
- 简单知识库问答

缺点：

- 检索不准
- 容易漏召回
- 对问题改写弱
- 没有评测容易幻觉

### 4.2 Advanced RAG

在 Naive RAG 上增强：

- 查询改写
- 意图识别
- Hybrid 检索
- 多路召回
- Rerank
- 上下文压缩
- 去重
- 自动评测

适合：

- 企业知识库
- 智能客服
- 测试知识库
- 产品助手

### 4.3 Agentic RAG

由 Agent 动态决定：

- 是否检索
- 检索哪个知识库
- 检索几次
- 是否调用其他工具
- 是否反思和修正

适合：

- 多步骤任务
- 复杂分析
- 自动生成报告
- 学习助手

### 4.4 Graph RAG

把文档抽取成：

- 实体
- 关系
- 路径

适合：

- 法律条文推理
- 金融风控
- 医疗知识关联
- 复杂依赖分析

第一阶段只讲概念和可视化，不做完整实现。

## 5. 第一版功能

第一版前端原型：

- RAG 学习地图
- RAG 流程步骤
- 文本切片演示
- Top K 检索演示
- RAG 类型切换
- 参数调节
- 结果解释

## 6. 第二版功能

接入真实后端：

- 上传文档
- 解析 TXT / MD / PDF / DOCX
- 文本切片
- SQLite 保存文档和切片
- Embedding 生成向量
- FAISS 或 Chroma 检索
- RAG 问答
- 引用来源

## 7. 第三版功能

做企业级优化：

- BM25
- Hybrid
- Rerank
- 无依据拒答
- 自动评测集
- 失败原因
- 参数对比
- 质量报告

## 8. 验收标准

RAG 实验室合格标准：

- 用户能看懂 RAG 每一步
- 用户能调整 chunk size、overlap、top k
- 用户能看到检索命中的片段
- 用户能理解 Embedding 和向量库的作用
- 系统能回答有依据问题
- 系统能拒答无依据问题
- 系统能保存评测结果
