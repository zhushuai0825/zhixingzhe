# 三大实验室关系说明

## 1. 总体关系

知行者 AI 实验室的核心由三个实验室组成：

```text
RAG 实验室
  -> 可视化实验台
  -> Agent 实验室
```

它们不是重复模块，而是 AI 应用从资料到行动的三个阶段。

一句话理解：

```text
RAG 实验室：建知识库
可视化实验台：查知识库
Agent 实验室：用知识库和工具完成目标
```

## 2. RAG 实验室

### 2.1 负责什么

RAG 实验室负责“资料怎么进入系统”。

完整流程：

```text
上传文档
  -> 解析文本
  -> 文本清洗
  -> 文本切片 chunk
  -> Embedding 向量化
  -> 存入向量数据库
  -> 建立索引
```

### 2.2 用户在这里学什么

用户要学会：

- 文档为什么需要解析
- 为什么长文档不能直接丢给大模型
- 什么是 chunk
- chunk size 和 overlap 有什么影响
- Embedding 为什么能表示语义
- 向量数据库里到底存什么
- FAISS / Chroma / Milvus 的作用是什么
- documents、document_chunks、chunk_vectors 分别存什么

### 2.3 页面重点

RAG 实验室页面要重点展示：

- 原始文档
- 解析后的文本
- 切片列表
- 每个切片的 token 数
- 每个切片对应的向量预览
- 数据库存储位置
- 向量索引建立过程

## 3. 可视化实验台

### 3.1 负责什么

可视化实验台负责“用户提问后怎么命中资料”。

完整流程：

```text
用户问题
  -> 问题向量化
  -> 向量数据库匹配
  -> Top K chunk 命中
  -> BM25 / Hybrid / Rerank 排序
  -> 选出上下文
  -> 给大模型生成回答
```

### 3.2 用户在这里学什么

用户要学会：

- 用户问题也会被 Embedding
- 向量检索如何比较问题和 chunk
- 为什么会命中这些 chunk
- Top K 是什么意思
- vector_score 是什么
- BM25 为什么能补充关键词检索
- Hybrid 为什么是工业界主流
- Rerank 为什么能提高最终排序
- 哪些片段会进入 Prompt 上下文

### 3.3 页面重点

可视化实验台页面要重点展示：

- 用户输入问题
- 问题向量预览
- 向量数据库匹配动画
- Top K 命中列表
- 每个 chunk 的分数
- Rerank 前后排序
- 最终进入 Prompt 的上下文

## 4. Agent 实验室

### 4.1 负责什么

Agent 实验室负责“AI 怎么从回答问题变成完成目标”。

普通 RAG 是：

```text
用户问一句
系统检索一次
模型回答一次
```

Agent 是：

```text
用户给一个目标
  -> AI 分析目标
  -> AI 决定是否需要查知识库
  -> AI 决定调用什么工具
  -> AI 读取工具结果
  -> AI 判断结果够不够
  -> 不够就继续检索或追问
  -> 够了就生成回答、任务、报告或下一步动作
```

### 4.2 用户在这里学什么

用户要学会：

- Agent 和普通 Chatbot 的区别
- Tool Calling 是什么
- 工具输入输出是什么
- Agent 为什么要保存状态
- Agent 为什么要有运行轨迹
- Agent 为什么要有限制和停止条件
- Agent 如何调用 RAG 检索工具
- Agent 如何生成任务
- Agent 如何反思结果是否可靠

### 4.3 页面重点

Agent 实验室页面要重点展示：

- 用户目标
- Agent 类型
- Plan
- Tool Call
- Observation
- Evaluation
- Action
- Reflection
- 工具输入
- 工具输出
- 失败原因

## 5. 为什么三个都要保留

因为它们分别回答三个问题：

### RAG 实验室回答：

```text
我的资料是怎么变成知识库的？
```

### 可视化实验台回答：

```text
我提问之后，系统是怎么找到相关资料的？
```

### Agent 实验室回答：

```text
AI 怎么利用知识库和工具帮我完成一个目标？
```

如果只做 RAG 实验室，用户只能学会建库。

如果只做可视化实验台，用户只能学会检索。

如果只做 Agent 实验室，用户会看不懂 Agent 调用的 RAG 工具从哪里来。

所以三者必须同时存在，但职责要清晰。
