# 可视化实验台需求

## 1. 模块定位

可视化实验台是知行者 AI 实验室的核心体验。

它要解决的问题是：

> 用户不只看到最终回答，还能看到数据如何进入系统、如何被切片、如何变成向量、如何存入向量数据库、用户问题如何命中文档片段、Agent 如何调用工具。

这个模块会把抽象的 RAG 和 Agent 变成可观察、可解释、可复盘的流程。

## 2. 是否还需要任务中心

需要保留，但不能作为主角。

任务中心的定位：

- 辅助学习
- 承接 Agent 生成的行动项
- 记录每个阶段要完成的练习
- 做学习闭环

任务中心不应该复杂化，不做项目管理系统。

第一版只保留：

- 待办
- 进行中
- 已完成
- 任务关联模块：RAG / Agent / API / 部署

## 3. 可视化目标

系统要让用户直观看到四件事：

1. 文档进入系统后存在哪里
2. 文档如何被切成 chunk
3. chunk 如何变成向量并进入向量数据库
4. 用户问题如何命中某些 chunk
5. Agent 如何调用 RAG 工具并继续决策

## 4. RAG 数据流可视化

### 4.1 文档入库流程

展示链路：

```text
上传文件
  -> documents 表
  -> document_chunks 表
  -> chunk_vectors 表
  -> FAISS / Chroma 索引
```

界面表现：

- 用流程图展示每个数据节点
- 鼠标点击节点后，右侧展示字段
- 动画高亮当前流转步骤
- 显示数据条数：文档数、切片数、向量数

### 4.2 数据库存储可视化

需要展示：

#### documents 表

- id
- file_name
- file_type
- file_size
- status
- created_at

#### document_chunks 表

- id
- document_id
- chunk_index
- content
- token_count

#### chunk_vectors 表

- chunk_id
- vector_json
- model_name
- provider
- dimensions

#### 向量索引

- index_type
- dimensions
- vector_count
- distance_metric

用户要能理解：

```text
原始文档存在 documents
文本切片存在 document_chunks
向量存在 chunk_vectors
向量库索引用于快速相似度检索
```

## 5. 检索命中可视化

用户输入问题后，展示：

```text
问题文本
  -> 问题 Embedding
  -> 向量检索
  -> Top K 命中
  -> BM25 分数
  -> Hybrid 分数
  -> Rerank 后顺序
  -> 进入 Prompt 的上下文
```

界面元素：

- 问题向量卡片
- 命中 chunk 列表
- 相似度进度条
- Top K 排名
- 分数拆解：vector_score / bm25_score / hybrid_score / rerank_score
- 被选入上下文的片段高亮

动画建议：

- 用户点击“运行检索”
- 问题卡片发光
- 箭头流向向量数据库
- 命中的 chunk 节点依次亮起
- Top K 结果飞入上下文区域

## 6. 向量空间可视化

为了让用户理解 Embedding，可以做二维近似图。

注意：

真实向量可能是 384 / 768 / 1024 维，前端不能直接看懂。

可视化方式：

- 用 PCA / t-SNE / UMAP 后端降维，前端展示 2D 点
- 第一版可以用模拟坐标
- 每个点代表一个 chunk
- 用户问题是一个特殊点
- 距离越近表示语义越接近

界面表现：

- 散点图
- 点大小表示匹配度
- 颜色表示文档来源
- 点击点显示 chunk 内容

## 7. Agent 运行可视化

展示 Agent 的执行轨迹：

```text
目标
  -> Plan
  -> Tool: knowledge_search
  -> Observation
  -> Evaluate
  -> Tool: task_generator
  -> Reflect
  -> Final
```

每个节点展示：

- 输入
- 输出
- 状态
- 耗时
- 是否成功
- 错误信息

动画建议：

- 当前节点高亮
- 工具调用时显示 loading
- 成功后显示绿色状态
- 失败后显示红色错误
- 反思节点用不同颜色标记

## 8. 推荐前端技术方案

### 8.1 第一版：原生 HTML + CSS + JavaScript + SVG

适合当前阶段。

用途：

- 做流程图
- 做节点高亮
- 做简单动画
- 做学习型原型

优点：

- 简单
- 不引入框架复杂度
- 方便快速修改

### 8.2 第二版：React + React Flow

适合做 RAG 流程图和 Agent 状态图。

用途：

- 节点
- 边
- 拖拽
- 缩放
- 自定义节点
- 展示 Agent 工作流

适合页面：

- RAG 数据流图
- Agent 工具调用图
- LangGraph 状态图

### 8.3 图谱可视化：Cytoscape.js

适合后期 Graph RAG。

用途：

- 实体关系图
- 多跳路径
- 知识图谱
- 节点拖拽
- 图算法

适合页面：

- Graph RAG 实验室
- 实体关系图
- 关系路径解释

### 8.4 图表统计：ECharts

适合做统计图。

用途：

- 评测通过率
- 检索分数对比
- 文档数量统计
- Top K 分数柱状图
- 向量空间散点图

适合页面：

- RAG 评测报告
- 检索质量报告
- 数据概览

### 8.5 自定义动画：D3.js

适合需要更自由的可视化时使用。

用途：

- 力导向图
- 自定义节点动画
- 向量点云
- 数据流动画

注意：

- D3 灵活但学习成本高
- 不建议第一版就大量使用

## 9. 推荐实施顺序

### V0

使用原生前端做：

- RAG 流程图
- 数据库存储关系图
- 检索命中演示
- Agent 运行轨迹

### V1

接入后端真实数据：

- documents
- chunks
- vectors
- search results
- agent runs

### V2

引入 ECharts：

- 检索分数图
- 评测通过率
- 向量二维散点图

### V3

如果切换 React，引入 React Flow：

- RAG 工作流编辑和展示
- Agent 节点运行图

### V4

做 Graph RAG 时，引入 Cytoscape.js：

- 实体关系图
- 多跳检索路径

## 10. 验收标准

可视化实验台合格标准：

- 用户能看到文档进入了哪些表
- 用户能看到 chunk 如何生成
- 用户能看到向量记录和索引关系
- 用户能看到问题命中了哪些 chunk
- 用户能看到每个命中结果的分数
- 用户能看到 Rerank 前后顺序变化
- 用户能看到 Agent 每一步调用了什么工具
- 用户能理解失败发生在哪一步
