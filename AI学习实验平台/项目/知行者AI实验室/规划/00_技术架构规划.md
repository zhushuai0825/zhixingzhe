# 技术架构规划

## 1. 总体架构

知行者 AI 实验室采用前后端分离架构。

```text
浏览器前端
  -> FastAPI 后端
  -> SQLite / PostgreSQL
  -> 文档解析服务
  -> Embedding 服务
  -> 向量数据库
  -> 大模型 API
  -> Agent 工作流
```

## 2. 前端

第一阶段使用：

- HTML
- CSS
- JavaScript

原因：

- 上手简单
- 可以快速验证产品结构
- 不被框架干扰

后期如果复杂度上升，可以迁移到：

- React
- Vue
- Vite

## 3. 后端

使用：

- Python
- FastAPI

原因：

- Python 是 AI 应用主流语言
- FastAPI 适合做 AI 系统接口
- 方便接 LangChain、LlamaIndex、向量库、文档解析库

## 4. 数据库

第一阶段正式使用：

- PostgreSQL

用途：

- 保存知识库
- 保存文档
- 保存切片
- 保存模型配置
- 保存 RAG 实验记录
- 保存 Agent 运行记录
- 保存任务

暂不使用 SQLite，避免学习路线和真实企业架构脱节。

## 5. 向量数据库

第一阶段正式使用：

- Chroma

用途：

- 保存 chunk embedding
- 建立向量索引
- 根据问题向量检索 Top K chunk_id
- 保存向量 metadata

架构关系：

```text
PostgreSQL：业务数据、文档、切片、任务、运行记录
Chroma：向量、向量索引、相似度检索
```

查询流程：

```text
用户问题
  -> 生成问题向量
  -> Chroma 检索 Top K chunk_id
  -> PostgreSQL 按 chunk_id 查询完整文本和文档信息
  -> 拼接 Prompt
  -> 大模型生成回答
```

后期可选：

- Qdrant
- Milvus
- pgvector

其中 pgvector 是 PostgreSQL 的向量扩展，但当前项目优先学习“独立向量数据库”，所以先用 Chroma。

## 6. 文档解析

使用：

- python-docx：Word
- pdfplumber：PDF
- PyPDF2：简单 PDF
- Markdown / TXT：直接读取
- OCR：后期处理图片和扫描 PDF

## 7. Embedding

第一阶段：

- 本地轻量 Embedding 或 API Embedding

可选：

- bge-small-zh
- bge-m3
- text-embedding 系列 API
- 通义 / OpenAI / 其他兼容接口

学习重点：

- 文本如何变成向量
- 向量维度是什么
- 相似度怎么计算
- 为什么向量能做语义检索

## 8. 检索

需要实现：

- 向量检索
- BM25 关键词检索
- Hybrid 检索
- Rerank

学习重点：

- 向量检索适合语义相似
- BM25 适合关键词精确匹配
- Hybrid 是工业界常用方案
- Rerank 用于提升最终证据质量

## 9. 大模型 API

要支持：

- DeepSeek
- 通义千问
- OpenAI
- 豆包
- Ollama 本地模型

统一抽象：

```text
provider
base_url
api_key
model
messages
temperature
```

## 10. Agent

第一阶段不用复杂框架，先手写流程：

```text
plan
retrieve
evaluate
act
reflect
stop
```

第二阶段接：

- Tool Calling
- JSON schema
- 工具参数校验

第三阶段接：

- LangGraph
- OpenAI Agents SDK

## 11. 部署

第一阶段：

- 本地运行

第二阶段：

- Docker
- Nginx
- 云服务器

第三阶段：

- 日志
- 监控
- 自动备份
- 权限系统
