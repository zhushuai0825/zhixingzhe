# RAG 实验室学习手册

这份文档用 0 基础方式解释当前系统里的 RAG。

## 1. RAG 是什么

RAG 是 Retrieval-Augmented Generation，中文常叫“检索增强生成”。

简单说：

```text
先查资料，再让大模型回答。
```

普通大模型问答的问题是：它可能凭记忆回答，也可能编。

RAG 的思路是：

1. 先把你的文档存进知识库。
2. 用户提问时，先从知识库找相关片段。
3. 把这些片段作为依据交给大模型。
4. 大模型只能围绕这些依据回答。
5. 回答时展示引用来源。

## 2. 知行者里的 RAG 流程

当前系统流程：

```text
上传文档
  -> 解析文本
  -> 文本切片
  -> 保存 PostgreSQL
  -> 生成 Embedding
  -> 写入 Chroma
  -> 用户提问
  -> 问题向量化
  -> 向量检索 + BM25 检索
  -> 融合排序
  -> 依据判断
  -> 大模型生成回答
  -> 展示引用来源
```

## 3. 文档解析

文档解析就是把文件里的文字读出来。

当前支持：

- `.txt`
- `.md`
- `.markdown`
- 可提取文字的 `.pdf`

暂不支持：

- 扫描版 PDF。
- 图片。
- Word。
- 音频。
- 视频。

扫描版 PDF 看起来是文字，但本质是图片，所以普通 PDF 解析工具读不到。后续需要 OCR。

## 4. 文本切片

大模型和向量库不适合直接处理特别长的文档，所以要切片。

切片就是把长文档拆成多个小片段。

例如：

```text
一份 10000 字文档
  -> chunk 1
  -> chunk 2
  -> chunk 3
  -> ...
```

当前系统支持两个核心参数：

- `chunk_size`：每个切片最大长度。
- `overlap`：相邻切片之间重复多少内容。

## 5. 为什么需要 overlap

如果没有重叠，重要句子可能刚好被切断。

例如：

```text
chunk 1 结尾：RAG 的关键价值是
chunk 2 开头：减少幻觉并展示引用来源
```

单独看 chunk 1 或 chunk 2 都不完整。

overlap 会让相邻 chunk 有一部分重复内容，减少信息断裂。

## 6. Embedding 是什么

Embedding 是把文本变成一串数字向量。

例如：

```text
"RAG 是什么"
  -> [0.12, -0.03, 0.45, ...]
```

向量的意义是：让机器可以计算文本之间的相似度。

人看到两句话知道它们意思接近，机器需要先把它们变成数字，才能比较。

## 7. 当前 Embedding 是什么

当前系统用的是 `local_hash_demo`。

它是本地学习版哈希向量。

优点：

- 不需要外部 API。
- 不花钱。
- 能离线学习完整流程。
- 可以真实写入 Chroma。

缺点：

- 不是真正语义模型。
- 对中文语义理解不如真实 Embedding。
- 适合学习流程，不适合生产。

后续可以替换成：

- bge-small-zh。
- bge-m3。
- 通义 Embedding。
- OpenAI Embedding。
- 其他兼容 Embedding API。

## 8. 向量数据库是什么

向量数据库负责保存向量，并支持相似度检索。

当前系统使用 Chroma。

Chroma 里存：

- chunk_id。
- chunk 向量。
- chunk 原文。
- metadata，例如知识库 ID、文档 ID、文件名。

用户提问时：

```text
问题 -> 问题向量 -> Chroma 查询 -> 返回最相似的 chunk
```

## 9. PostgreSQL 和 Chroma 的区别

PostgreSQL 存业务数据：

- 知识库。
- 文档。
- 切片。
- RAG 运行记录。
- Agent 运行记录。

Chroma 存向量数据：

- embedding。
- 向量索引。
- 相似度检索。

一句话：

```text
PostgreSQL 管业务记录，Chroma 管向量检索。
```

## 10. 向量检索

向量检索适合找语义相似的内容。

例如用户问：

```text
怎么让 AI 不乱编？
```

向量检索可能命中：

```text
RAG 的关键价值是减少模型幻觉，让回答基于用户自己的知识库。
```

虽然字面不完全一样，但语义接近。

## 11. BM25 关键词检索

BM25 是经典关键词检索算法。

它适合精确词匹配。

例如用户问：

```text
chunk_size 是什么？
```

BM25 会更重视包含 `chunk_size` 的片段。

## 12. Hybrid RAG

Hybrid RAG 是向量检索 + 关键词检索融合。

当前系统学习版公式：

```text
hybrid_score = vector_score * 0.7 + keyword_score * 0.3
```

为什么要 Hybrid：

- 向量检索适合理解语义。
- BM25 适合抓关键词。
- 两者结合更稳。

企业知识库里，Hybrid RAG 很常见。

## 13. Top K 是什么

Top K 表示取最相关的前 K 条。

例如 `top_k = 5`：

```text
取相关度最高的 5 个 chunk
```

Top K 太小，可能漏掉证据。

Top K 太大，可能带入噪声，让大模型分心。

## 14. 最低相关度 min_score

`min_score` 是证据门槛。

如果最高命中分数太低，系统应该拒答。

原因：

```text
没有可靠依据时，宁愿说不知道，也不要编。
```

这就是 RAG 系统比普通聊天更可靠的关键。

## 15. 引用来源

引用来源告诉你答案基于哪些 chunk。

一个好 RAG 系统必须能回答：

- 引用了哪个文件？
- 引用了哪个 chunk？
- 原文是什么？
- 相关度多少？
- 是向量命中还是关键词命中？

当前系统已经支持在 Agent 最终结果里展开引用原文。

## 16. 怎么判断 RAG 做得好

看四个指标：

- 召回是否相关。
- 回答是否只基于证据。
- 引用是否能支撑答案。
- 证据不足时是否拒答。

测试问题可以分三类：

- 文档里明确有答案的问题。
- 文档里只有部分答案的问题。
- 文档里完全没有答案的问题。

## 17. 下一步该学什么

在当前系统基础上，建议下一步学习：

1. 真实 Embedding 模型。
2. Rerank 重排序。
3. RAG 自动评测。
4. OCR 和图片文档处理。
5. 多轮 Conversational RAG。
6. Agentic RAG。
