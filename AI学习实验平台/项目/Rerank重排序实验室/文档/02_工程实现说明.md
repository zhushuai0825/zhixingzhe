# 工程实现说明

## 1. 当前版本是什么

当前 Rerank 实验室是纯前端教学版。

它不调用真实 rerank API，而是用可解释规则模拟真实流程：

```text
query
  -> 向量召回模拟分
  -> 词面匹配分
  -> 可回答性分
  -> 来源质量分
  -> 最终 rerank 分
```

这样设计的原因是：学习阶段先把过程看明白，再接真实模型。

## 2. 数据结构

每个 chunk 类似这样：

```js
{
  id: "c1",
  title: "Rerank 的作用",
  source: "rerank_intro.md",
  quality: 0.96,
  text: "Rerank 会对向量库初步召回的候选 chunk 重新排序..."
}
```

字段含义：

- `id`：chunk 唯一标识。
- `title`：页面展示标题。
- `source`：来源文件。
- `quality`：模拟来源质量。
- `text`：chunk 正文。

## 3. 向量召回怎么模拟

教学版没有真正生成 embedding，而是用关键词重合模拟向量召回：

```text
vectorScore = query 和 chunk 的关键词重合程度 + 标题加权 + 来源质量轻微加权
```

它的目标不是替代真实 embedding，而是让你看到“召回阶段可能把相似主题片段先捞出来”。

## 4. Rerank 分怎么模拟

Rerank 分由四部分组成：

```text
rerankScore =
  vectorScore 权重
  + 词面匹配权重
  + 可回答性权重
  + 来源质量权重
```

页面提供三种策略：

- 平衡：相关性、可回答性、质量都看。
- 严格：更重视能不能直接回答问题。
- 语义：更接近向量召回结果。

## 5. 可视化怎么读

排序可视化中：

- 左侧：向量库初筛顺序。
- 右侧：Rerank 后顺序。
- 连线：同一个 chunk 的名次变化。
- 上升：reranker 认为它更能回答问题。
- 下降：主题相关但证据价值不足。

## 6. 以后怎么接真实模型

后续可以新增后端接口：

```text
POST /api/rerank
```

请求：

```json
{
  "query": "Rerank 在 RAG 里起什么作用？",
  "candidates": [
    { "id": "c1", "text": "..." }
  ],
  "topN": 3
}
```

返回：

```json
{
  "results": [
    {
      "id": "c1",
      "score": 0.92,
      "reason": "直接解释了 Rerank 的作用"
    }
  ]
}
```

真实模型可选：

- 本地 BGE Reranker。
- Cohere Rerank API。
- Jina Reranker API。
- 大模型打分。

## 7. 当前验收标准

- 修改问题后，召回和重排结果会变化。
- 修改策略后，排序可能变化。
- 修改 Top K、Top N 后，候选数量和最终证据数量会变化。
- 点击候选后，详情区能解释分数来源。
- 可视化能展示排序前后名次移动。
